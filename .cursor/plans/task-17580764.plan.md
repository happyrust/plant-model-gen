# 解析任务进度服务：REST + WebSocket 补齐方案

## 后端（Rust / Axum）

- 新增 REST 端点
- GET `/api/tasks/{id}/progress`：返回 `{ progress: TaskProgress, status: TaskStatus }`
- GET `/api/tasks/status`：返回任务队列快照（运行中/等待/完成数量、最近更新时间）
- 新增 WebSocket 推送
- 端点：`/api/tasks/ws`（可选带参：`?id=<taskId>` 仅推送单任务，否则推送所有任务更新）
- 事件：`{"type":"task_update"|"queue_update","data":TaskInfo 或 摘要}`
- 推送集成
- 在 `update_progress(...)` 和任务状态变更处（start/complete/fail/cancel）广播事件
- 广播实现：`tokio::sync::broadcast`（全局 sender 存于 `AppState`）
- 关键文件与修改点
- `src/web_ui/mod.rs`：注册新路由
- `src/web_ui/handlers.rs`：实现 `get_task_progress`、`get_tasks_status`、集成广播；复用现有 `get_task`
- `src/web_ui/models.rs`：新增 `TaskUpdateMessage`/`QueueStatus` 等序列化类型

## 前端（Next.js）

- 直连后端 API（移除对 `app/api/tasks/[taskId]/progress` 代理的依赖）
- 调用 `GET ${API_BASE_URL}/api/tasks/{id}/progress`、`GET ${API_BASE_URL}/api/tasks/status`
- 新增 WebSocket 客户端
- 地址：`ws(s)://${API_BASE_URL}/api/tasks/ws`
- 提供 `useTaskProgressWS(taskId?)` Hook：自动区分单任务/全量推送
- UI 对接
- `TaskStatusCard`/`TaskQueueMonitor` 优先使用 WS 实时数据；降级轮询 REST
- 更新 `lib/api/task-monitor.ts`：新增直接请求的函数；标记旧代理 unused

## 兼容性与配置

- CORS：后端允许来自前端域名的 WS/REST 访问
- 环境变量：`NEXT_PUBLIC_API_BASE_URL` 用于 REST；WS 以同源或从 BASE_URL 推导

## 验证

- 启动长任务（解析），观察前端实时进度；断网/重连后自动恢复订阅
- 回归：历史任务查询、日志分页不受影响
