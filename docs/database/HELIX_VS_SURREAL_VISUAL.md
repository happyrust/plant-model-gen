# HelixDB vs SurrealDB 递归查询可视化对比

## 查询方式对比

### SurrealDB (多次查询模式)

```
客户端                          数据库
  |                               |
  |---查询节点 1001 的子节点----->|
  |<------返回 [1002, 1003]------|  (往返 1)
  |                               |
  |---查询节点 1002 的子节点----->|
  |<------返回 [1004, 1005]------|  (往返 2)
  |                               |
  |---查询节点 1003 的子节点----->|
  |<------返回 [1006]-------------|  (往返 3)
  |                               |
  |---查询节点 1004 的子节点----->|
  |<------返回 [1007, 1008]------|  (往返 4)
  |                               |
  ... 继续查询每个节点 ...
  |                               |
  总计: N 次往返（N = 节点数）
```

### HelixDB (单次查询模式)

```
客户端                          数据库
  |                               |
  |---Cypher: MATCH递归查询------>|
  |                               |
  |                               | [数据库内部处理]
  |                               | - 使用图遍历算法
  |                               | - 优化路径查找
  |                               | - 一次性获取所有结果
  |                               |
  |<---返回所有匹配节点----------|  (往返 1)
  |                               |
  总计: 1 次往返
```

## 查询代码对比

### 示例: 查找 Site 下所有深度为 2-4 的 PIPE 节点

#### SurrealDB 实现 (约 40 行代码)

```rust
async fn find_pipes(
    db: &DB,
    site_refno: RefU64,
) -> Result<Vec<RefU64>> {
    let mut pipes = Vec::new();
    let mut queue = VecDeque::new();
    queue.push_back((site_refno, 0));
    let mut visited = HashSet::new();

    while let Some((refno, depth)) = queue.pop_front() {
        if depth > 4 || visited.contains(&refno) {
            continue;
        }
        visited.insert(refno);

        // 查询类型
        let type_name = db.get_type_name(refno).await?;

        // 检查深度和类型
        if depth >= 2 && depth <= 4 && type_name == "PIPE" {
            pipes.push(refno);
        }

        // 继续遍历
        if depth < 4 {
            let children = db.get_children(refno).await?;
            for child in children {
                queue.push_back((child, depth + 1));
            }
        }
    }

    Ok(pipes)
}

// 查询次数: O(N) 其中 N 是所有访问的节点数
// 网络往返: O(N)
```

#### HelixDB 实现 (1 行 Cypher)

```rust
async fn find_pipes(
    helix: &HelixDB,
    site_refno: RefU64,
) -> Result<Vec<RefU64>> {
    let query = format!(
        "MATCH (site:Element {{refno: $site_refno}})
               -[:HAS_CHILD*2..4]->(pipe:Element {{type_name: 'PIPE'}})
         RETURN pipe.refno"
    );

    let result = helix.execute(query).await?;
    Ok(parse_refnos(result))
}

// 查询次数: 1
// 网络往返: 1
```

**代码简洁性**: HelixDB 代码量减少 **95%**

## 实际应用场景对比

### 场景 A: 生成 XKT 模型时查询几何节点

**需求**: 从 Site 递归查询所有几何节点（PRIM, LOOP, CATA 等）

#### SurrealDB 流程:
```
1. 查询 Site → 1 次
2. 获取 Site 的子节点 → 1 次
3. 对每个子节点:
   - 查询类型 → N 次
   - 如果匹配几何类型，查询几何参数 → M 次
   - 如果有子节点，递归查询 → 递归深度 × 节点数
4. 处理负实体（再次递归）→ 再次递归

总查询: 数百到数千次
总耗时: 数秒到数十秒
```

#### HelixDB 流程:
```
1. 单次 Cypher 查询获取所有几何节点:
   MATCH (site:Site {refno: $site})
         -[:HAS_CHILD*]->(geo:Element)
   WHERE geo.type_name IN ['PRIM', 'LOOP', 'CATA', ...]
   RETURN geo

总查询: 1 次
总耗时: 毫秒级
```

**性能提升**: 100-1000x

---

### 场景 B: 查找桥架支撑关系

**需求**: 查找 BRAN → SCTN → 支撑结构 的关系

#### SurrealDB 流程:
```
1. 查询 BRAN 的所有 SCTN → 多次查询
2. 对每个 SCTN:
   - 查询空间索引找附近节点 → N 次
   - 验证每个候选节点的类型 → M 次
   - 检查几何关系 → P 次

总查询: N × M × P 次（可能数千次）
```

#### HelixDB 流程:
```
1. 单次图查询:
   MATCH (bran:BRAN)-[:HAS_CHILD*]->(sctn:SCTN)
         -[:SPATIAL_NEAR]->(support:Structure)
   WHERE support.type_name IN ['BEAM', 'COLUMN']
   RETURN sctn, support

总查询: 1 次
```

**性能提升**: 1000x+

## 总结

### HelixDB 在递归遍历的核心优势:

```
┌─────────────────────────────────────────────────────────┐
│  1. 查询次数:    1 次 vs N 次      (100-1000x 减少)    │
│  2. 网络往返:    1 次 vs N 次      (延迟降低 99%)      │
│  3. 代码复杂度:  1 行 vs 40+ 行    (简洁 95%)         │
│  4. 数据库优化:  原生图算法支持    (算法级优化)        │
│  5. 可维护性:    声明式查询        (易于理解和维护)     │
│  6. 扩展性:      适应大规模数据    (性能稳定)          │
└─────────────────────────────────────────────────────────┘
```

### 适用场景建议:

| 查询类型 | 推荐数据库 | 原因 |
|---------|-----------|------|
| 单节点属性查询 | SurrealDB | 简单查询，性能相当 |
| 深度 > 2 的递归 | HelixDB | 网络往返优势明显 |
| 路径查询 | HelixDB | 内置算法优化 |
| 复杂模式匹配 | HelixDB | 声明式查询强大 |
| 图关系分析 | HelixDB | 原生图数据库 |
| 空间查询 | 专用索引 | 使用 R-tree 等 |
