# 自主可控数据库比选报告

## 1. 项目背景与系统需求
- `gen-model` 需要支撑 PDMS/E3D 数据的层级遍历、元素关系与增量写入，当前默认数据库接口已切换为 SurrealDB，强调对复杂图关系与动态属性的支持。参见 `SurrealDB迁移进度报告.md:10-65`。
- `PdmsDataInterface` 扩展了深度子节点查询、命名属性映射、实例数据写入等接口，上层 `gen_geos` 依赖这些能力进行几何生成。参见 `SurrealDB适配gen_geos分析报告.md:5-33`。
- 现有关系型实现（以 MySQL 为代表）存在严重的 N+1 查询、递归开销和结构化 schema 约束，难以满足深度遍历与动态属性场景。参见 `E3D层级查询接口优化分析.md:9-140`。

基于以上背景，系统对数据库的核心诉求包括：
1. **原生图关系与多模型能力**：需要以节点+边方式表达 `pe_owner`、`owns` 等拓扑，并支持单条查询完成多层遍历。
2. **动态属性与 JSON 结构友好**：`named_attributes`、`implicit_attributes` 等需要灵活扩展，减少频繁 DDL。
3. **高并发批量写入与事务**：增量生成和实例写入需要原子操作、批处理与实时反馈。
4. **多协议接入**：现有架构通过 WebSocket/HTTP 调 SurrealDB，期望数据库原生支持长连接与事件推送。

## 2. 候选自主可控数据库产品评估

### 2.1 openGauss
**产品概述**：开源、国产化的企业级关系型数据库，兼容 PostgreSQL 生态，主打金融、电信等传统 OLTP/OLAP 场景。

**优势**
- 具备成熟的 SQL 能力、分布式事务与企业运维工具；
- 支持 JSONB、递归 CTE，可在一定程度上描述树形结构；
- 生态完善，安全合规证书齐全，便于在国产化环境中部署。

**与项目需求的差距**
1. **缺乏原生图数据模型**：openGauss 仍以关系表为核心，复杂层级遍历需借助递归 CTE 与多次 JOIN，难以达到 SurrealDB 单次图查询的效率，仍会出现类似 MySQL 的 N+1 问题，在 `E3D层级查询接口优化分析.md:21-58` 中暴露的瓶颈难以根除。
2. **动态属性支持有限**：JSONB 字段虽可存储动态属性，但缺少细粒度索引与联合查询优化，`named_attributes` 等高变动字段会导致查询计划复杂度飙升，难以在毫秒级完成多条件过滤。
3. **多协议能力不足**：openGauss 原生接口以 SQL/TCP 为主，不提供 WebSocket 级的实时推送，需要额外开发中间件才能与现有 `RuntimeDatabaseAdapter` 的实时性契合。
4. **图算法扩展成本高**：若要支持类似 `query_multi_deep_versioned_children_filter_inst` 的接口，需要在数据库侧建立大量视图或外部存储过程，开发与运维负担大。

### 2.2 达梦 DM8
**产品概述**：达梦数据库是国内老牌自主可控关系型数据库，提供事务、并行计算、安全审计等企业特性，广泛应用于政务、金融、电力等领域。

**优势**
- 完整的事务、安全与多活方案，满足等级保护、审计合规要求；
- 兼容 Oracle/SQL 标准，具备较强的 SQL 与存储过程能力；
- 在国产化改造项目中有大量成功案例，生态与服务成熟。

**与项目需求的差距**
1. **仅支持关系模型**：DM8 缺乏图数据库特性，对树/图结构只能通过层次查询或嵌套子查询实现，会重现 `E3D层级查询` 文档中描述的递归查询与重复遍历问题，不适合 `gen_geos` 的深度图遍历。
2. **动态属性扩展性弱**：虽然 DM8 提供大字段与 JSON 支持，但索引、函数生态有限，`named_attributes`、`implicit_attributes` 等高频变动字段需要频繁拆表或使用外部缓存，违背“一条查询完成组合过滤”的需求。
3. **生态封闭、开发成本高**：为支撑复杂 SurrealQL 查询，需要编写大量 PL/SQL 存储过程或外部服务，且缺少与 Rust/sqlx 的成熟异步驱动，无法直接复用 `PdmsDataInterface` 既有的异步接口设计。
4. **实时写入与增量同步难度大**：DM8 的批量写入以传统批处理为主，若要实现 WebSocket 场景下的实时推送与增量事务，需要叠加消息队列或 CDC 组件，整体复杂度大幅高于当前 SurrealDB 实现。

### 2.3 人大金仓 KingbaseES（可选对比）
**产品概述**：KingbaseES 基于 PostgreSQL 深度定制，主打自主可控与国产软硬件适配，覆盖政府、能源等行业。

**与项目需求的主要差距**
- 与 openGauss 类似，仍是纯关系模型，对 `pe_owner` 图拓扑需要多次 JOIN 和 CTE，难以支持快速深度遍历；
- JSON/JSONB 支持程度有限，缺乏属性级索引与数组操作优化，面对 `named_attributes` 这类高维数据容易造成性能瓶颈；
- 缺少原生 WebSocket/事件流机制，无法直接满足现有系统对实时增量写入与回调的需求。

## 3. 现有市场方案无法满足系统需求的原因总结
1. **图数据缺口**：国产关系型数据库普遍缺少原生图/多模型能力，无法像 SurrealDB 一样在单次查询中完成多层遍历与过滤，难以支撑 `SurrealDB适配gen_geos分析报告.md:48-95` 中提到的接口需求。
2. **动态属性扩展受限**：系统依赖的命名属性、隐含属性需要弹性 schema，而主流自主可控关系数据库仍以固定字段为核心，JSON 支持不足以覆盖高并发读写，易造成性能退化。
3. **实时与增量流程复杂**：现有产品的主流使用方式侧重批量事务，缺少 WebSocket/流式 API，必须叠加自研缓存、消息队列弥补，整体架构复杂度与运维成本高。
4. **研发与运维投入高**：为了达到项目要求，需要大量存储过程、物化视图以及手工优化，违背当前团队希望通过统一接口迅速落地的目标。

## 4. 建议
1. 保持 SurrealDB 作为核心模型数据存储，继续补齐 `SurrealDBManager` 的剩余接口并完善测试，确保与 `PdmsDataInterface` 完全匹配。
2. 若需满足国产化合规，可将 openGauss/DM8 作为结构化数据的补充存储，继续承担项目、用户等传统业务数据，减少与核心模型查询的耦合。
3. 编写对照测试与迁移脚本，确保 SurrealDB 与传统关系库之间的数据一致性，为未来可能的多数据库混合部署留出接口。
