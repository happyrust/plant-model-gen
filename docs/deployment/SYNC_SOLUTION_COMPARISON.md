# 异地协同解决方案对比

## 三种方案对比

### 方案 1：纯 MQTT + 文件同步（现有）

**架构**:
```
服务器 A ←─── MQTT Broker ───→ 服务器 B
    ↓                              ↓
  发布文件                       订阅下载
    ↓                              ↓
  HTTP 文件服务器 ←─────────→ HTTP 客户端
```

**优点**:
- ✅ 已经实现，代码成熟
- ✅ 灵活的发布/订阅模式
- ✅ 适合传输大文件
- ✅ 实时性好（< 100ms）

**缺点**:
- ❌ 配置信息不自动同步（需手动配置 MQTT）
- ❌ 需要手动输入文件服务器地址
- ❌ 没有统一的配置视图
- ❌ 复杂度高（MQTT + HTTP + 文件监控）

**适用场景**:
- 文件同步（数据库文件、资源包）
- 实时状态更新
- 任务协调

---

### 方案 2：LiteFS 数据库同步（推荐）

**架构**:
```
服务器 A (主节点)           服务器 B (副本)
    ↓                              ↓
  LiteFS                        LiteFS
    ↓                              ↓
deployment_sites.sqlite ────→ deployment_sites.sqlite
  (自动同步配置)                (只读副本)
```

**优点**:
- ✅ **零配置同步**：在任意节点创建环境，自动同步到所有节点
- ✅ **实时同步**：秒级延迟（< 1s）
- ✅ **零代码修改**：只需修改数据库路径
- ✅ **高可用性**：主节点故障自动切换
- ✅ **统一视图**：所有节点看到相同的配置

**缺点**:
- ❌ 需要部署 LiteFS daemon
- ❌ 写操作只能在主节点
- ❌ 不适合大文件同步

**适用场景**:
- 配置信息同步（环境、站点、部署配置）
- 元数据同步
- 小型数据库（< 10GB）

---

### 方案 3：LiteFS + MQTT 混合架构（终极方案）

**架构**:
```
┌────────────────────────────────────────────┐
│          配置层（LiteFS）                   │
│  deployment_sites.sqlite 自动同步          │
│    - remote_sync_envs（环境配置）          │
│    - remote_sync_sites（站点配置）         │
│    - deployment_sites（部署配置）          │
└────────────────────────────────────────────┘
              ↕ 秒级同步
┌────────────────────────────────────────────┐
│         数据层（MQTT + HTTP）               │
│  - 数据库文件传输                          │
│  - 大型资源包同步                          │
│  - 实时状态更新                            │
└────────────────────────────────────────────┘
```

**优点**:
- ✅ **最佳实践**：配置自动同步 + 文件灵活传输
- ✅ **零配置部署**：在任意服务器上启动即可使用
- ✅ **完整功能**：配置同步 + 文件同步 + 实时更新
- ✅ **性能优化**：各层使用最适合的技术

**缺点**:
- ⚠️ 复杂度较高（两套系统）
- ⚠️ 需要维护 LiteFS + MQTT

**适用场景**:
- 生产环境大规模部署
- 需要完整异地协同功能
- 配置 + 数据都需要同步

---

## 详细对比表

| 维度 | 纯 MQTT | 纯 LiteFS | LiteFS + MQTT |
|-----|---------|-----------|---------------|
| **配置同步** | 手动 | 自动 ⭐ | 自动 ⭐ |
| **数据同步** | 灵活 ⭐ | 不适合 | 灵活 ⭐ |
| **实时性** | < 100ms ⭐ | < 1s | 综合 ⭐ |
| **一致性** | 最终一致 | 强一致 ⭐ | 分层一致 ⭐ |
| **部署复杂度** | 中等 | 中等 | 较高 |
| **代码修改** | 较多 | 最少 ⭐ | 少 |
| **可扩展性** | 好 ⭐ | 一般 | 好 ⭐ |
| **故障恢复** | 手动 | 自动 ⭐ | 自动 ⭐ |
| **学习成本** | 中等 | 低 ⭐ | 中等 |

## 使用场景推荐

### 场景 1: 快速实现配置同步
**推荐**: LiteFS（方案 2）

**理由**:
- 最快实施（4.5 天）
- 零配置体验
- 满足配置同步需求

**实施**:
```bash
# 1. 部署 LiteFS
docker-compose up -d

# 2. 修改数据库路径
deployment_sites_sqlite_path = "/litefs/deployment_sites.sqlite"

# 3. 完成！
```

---

### 场景 2: 只需要文件同步
**推荐**: 纯 MQTT（方案 1）

**理由**:
- 已有实现，无需额外开发
- 适合大文件传输
- 灵活的发布订阅

**使用**:
- 使用现有的 `remote_sync_handlers.rs`
- 配置 MQTT broker
- 使用文件监控和传输功能

---

### 场景 3: 生产环境完整方案
**推荐**: LiteFS + MQTT（方案 3）

**理由**:
- 配置自动同步（LiteFS）
- 数据灵活传输（MQTT）
- 高可用 + 高性能

**架构**:
```
[配置] → LiteFS    → 自动同步到所有节点
[数据] → MQTT      → 发布订阅传输
[文件] → HTTP      → 内置文件服务器
```
