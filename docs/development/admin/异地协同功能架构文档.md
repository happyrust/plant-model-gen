## 1. 功能概述

异地协同是 plant-model-gen 的核心功能模块之一，解决多地部署站点之间的 **增量数据同步** 问题。

核心场景：多个物理站点（如北京、上海工厂）各自运行独立的 web_server + SurrealDB，通过 MQTT 消息通道 + HTTP 文件服务器实现 PDMS 增量数据的跨站点同步。

### 5.1 页面组件

```
├── GroupLogsPanel.vue         日志表格 + 筛选器
├── CollaborationEnvDrawer.vue 协同组创建/编辑侧栏
└── CollaborationSiteDrawer.vue 站点创建/编辑侧栏
```

### 5.2 状态管理 (Pinia)

`useCollaborationStore` 是功能核心，管理：

- **原始数据**: envs, sites, logs, dailyStats, flowStats, metadataBySiteId
- **运行态**: runtimeStatus, runtimeConfig
- **诊断态**: envDiagnostics, siteDiagnosticsById
- **UI 状态**: loading, refreshing, activating, deleting, importing, applying, stopping
- **派生计算**:
  - `groups` → 从 envs + runtime 计算出列表视图模型
  - `siteCards` → 从 sites + metadata + diagnostics 计算站点卡片
  - `insightsSummary` → 从 dailyStats + flowStats + logs 计算洞察
  - `overviewMetrics` → 4 个核心指标卡片
  - `effectiveState` → 运行时绑定状态语义化

### 5.3 关键交互流程

1. **初始化**: `initialize()` → 加载 envs → 自动选择第一个 → 加载 detail
2. **切换协同组**: `selectEnv()` → 加载 sites + logs + stats + metadata
3. **激活**: `activateSelectedEnv()` → POST activate → 刷新 runtime status
4. **诊断**: `runDiagnostics()` → 并行测试 MQTT + HTTP + 各站点
5. **导入**: `importEnvFromDbOption()` → 从当前配置创建协同组

## 6. 运行时架构（后端）

### 6.1 RuntimeState

```rust
pub struct RuntimeState {
    pub env_id: String,           // 当前激活的协同组
    pub mgr: Arc<AiosDBManager>,  // 数据管理器
    pub watcher_handle: JoinHandle,  // 文件监听任务
    pub mqtt_handle: JoinHandle,     // MQTT 订阅任务
}
```

全局单例 `REMOTE_RUNTIME: Lazy<RwLock<Option<RuntimeState>>>`

### 6.2 激活流程

1. 从 SQLite 读取协同组配置
2. 将配置写入 DbOption.toml
3. 初始化 `AiosDBManager`
4. 启动 `PdmsWatcher` 文件监听
5. 启动 MQTT 订阅（`poll_sync_e3d_mqtt_events_with_backoff`）
6. 写入全局 REMOTE_RUNTIME

### 6.3 MQTT 同步流程

1. MQTT broker 广播 `SyncE3dFileMsg`
2. 本地 subscriber 接收消息
3. 解析 file_names + file_server_host
4. 通过 HTTP 下载增量文件
5. `delta_clone_remotes` 写入本地
6. SurrealDB 数据更新

## 7. 配置依赖

### 8.1 整体评价

异地协同功能 **整体架构清晰、分层合理**，前后端职责明确：

- **前端**: Store 逻辑集中、派生计算充分，视图组件纯展示、通过 props/events 驱动，状态机语义化做得好（effectiveState 多分支覆盖）
- **后端**: SQLite CRUD 参数化查询安全，错误处理使用 StatusCode 映射，运行时管理用 RwLock 单例

**质量亮点**:

- 类型定义（`collaboration.ts`）覆盖全面，354 行纯类型文件
- 诊断系统设计完善：MQTT/HTTP/站点三级诊断，带延迟测量和时间戳
- 元数据获取三级回退策略（remote_http → cache → local_path）
- Store 中 `effectiveState` 计算覆盖了 5 种运行态组合

### 8.2 发现的问题

#### P1 - 高优先级


| #   | 问题                  | 位置                                      | 说明                                                                                                         |
| --- | ------------------- | --------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| 1   | **每次请求重新打开 SQLite** | `remote_sync_handlers.rs:open_sqlite()` | 每个 API 调用都重新打开连接并执行 `CREATE TABLE IF NOT EXISTS` + `ALTER TABLE`，对高频请求（如日志查询）存在性能浪费。建议使用连接池或 AppState 共享连接 |
| 2   | **删除协同组未检查运行态**     | `delete_env()`                          | 如果正在运行的协同组被删除，REMOTE_RUNTIME 中仍持有已删除的 env_id，可能导致状态不一致                                                     |
| 3   | **前端 store 过大**     | `stores/collaboration.ts` (1200+ 行)     | 单个 store 文件承载了全部业务逻辑，建议按领域拆分（如 diagnostics、logs、runtime 独立）                                                |


#### P2 - 中优先级


| #   | 问题                                   | 位置                        | 说明                                             |
| --- | ------------------------------------ | ------------------------- | ---------------------------------------------- |
| 4   | **错误信息被吞没**                          | `remote_sync_handlers.rs` | 大部分 handler 用 `.map_err(                       |
| 5   | **SQL 注入风险低但 ON DELETE CASCADE 不可靠** | `delete_env()`            | 注释说明 CASCADE 并非总是启用而显式删除 sites，但未删除关联的 logs 记录 |
| 6   | **诊断并发无限制**                          | `runDiagnostics()`        | 对所有站点并行发起 HTTP 诊断，站点数量多时可能产生大量并发请求             |
| 7   | **缺少输入校验**                           | `create_site()`           | 仅校验 name 非空，`http_host` 未校验格式（可能导致后续诊断或文件代理异常） |


#### P3 - 低优先级 / 建议


| #   | 问题                             | 位置            | 说明                                                     |
| --- | ------------------------------ | ------------- | ------------------------------------------------------ |
| 8   | API 路径命名不统一                    | 路由配置          | `/api/remote-sync/`* 与前端概念"协同(collaboration)"不匹配，可考虑统一 |
| 9   | 日志分页缺少 cursor-based pagination | `list_logs()` | 目前用 offset/limit，数据量大时性能下降                             |
| 10  | 前端缺少乐观更新                       | Store CRUD 操作 | 每次操作后完整重新加载列表，小概率可感知闪烁                                 |
| 11  | TypeScript 类型与后端结构体手工同步        | 前后端类型定义       | 无自动生成机制，后端加字段时前端可能遗漏                                   |


### 8.3 安全审查


| 检查项    | 结果                                     |
| ------ | -------------------------------------- |
| SQL 注入 | ✅ 安全 — 全部使用参数化查询                       |
| XSS    | ✅ 安全 — Vue 模板自动转义                      |
| CSRF   | ⚠️ 未见 CSRF token 机制，但 admin 后台通常为内网使用  |
| 认证授权   | ⚠️ 所有 /api/remote-sync/* 无认证中间件，依赖网络隔离 |
| 文件路径遍历 | ⚠️ `serve_site_files` 代理远端文件，需确认路径过滤   |

### 8.4 改进建议优先排序

1. **短期**（低成本高收益）：
  - 删除协同组时检查并停止关联运行时
  - 为 API 错误响应附带具体错误信息
2. **中期**（需要规划）：
  - SQLite 连接复用（连接池或 AppState 注入）
  - Store 按领域拆分
  - 添加 /api/remote-sync 认证中间件
3. **长期**（架构演进）：
  - API 路径统一为 /api/collaboration
  - 前后端类型自动同步机制
  - cursor-based 日志分页

## 9. Store 拆分方案（P3 参考）

当前 `stores/collaboration.ts` 约 1200 行，建议按领域拆分为：


| 文件                                    | 职责                | 包含内容                                                                    |
| ------------------------------------- | ----------------- | ----------------------------------------------------------------------- |
| `stores/collaboration/index.ts`       | 主 store，编排子 store | envs, selectedEnvId, initialize, selectEnv                              |
| `stores/collaboration/runtime.ts`     | 运行时管理             | runtimeStatus, runtimeConfig, activate, stop, effectiveState            |
| `stores/collaboration/diagnostics.ts` | 诊断                | envDiagnostics, siteDiagnosticsById, runDiagnostics, diagnosticsSummary |
| `stores/collaboration/logs.ts`        | 日志与统计             | logs, dailyStats, flowStats, logFilters, insightsSummary                |
| `stores/collaboration/sites.ts`       | 站点管理              | sites, metadataBySiteId, siteCards, CRUD                                |
| `stores/collaboration/helpers.ts`     | 纯函数               | parseNumberList, normalize*, buildStatusLabel, diagnostic*              |


拆分原则：

- 子 store 之间通过参数传递数据，避免循环依赖
- 主 store 负责初始化和编排
- helpers 提取为纯函数，不依赖 reactive 状态
- 视图组件 import 不变（仍从 `useCollaborationStore` 获取所有内容）
