# 本机站点编排 (Managed Sites) 功能架构文档

> **文档版本**: 2026-04-16
>
> **入口**: `/admin/#/sites`
>
> **定位**: 在当前机器上为项目生成独立运行目录、独立 DbOption、独立 SurrealDB 端口、独立 web_server 进程的编排工具

## 1. 功能概述

| 能力 | 状态 | 说明 |
|------|------|------|
| 站点 CRUD | ✅ | 创建/编辑/删除管理站点 |
| 解析项目数据 | ✅ | 异步解析 PDMS 数据 |
| 启动/停止站点 | ✅ | 拉起 SurrealDB + web_server 进程 |
| 运行态查看 | ✅ | CPU/内存/进程状态实时监控 |
| 日志查看 | ✅ | 解析日志/数据库日志/站点日志 |
| 机器资源监控 | ✅ | CPU/内存/磁盘/数据目录大小 |
| 风险评估 | ✅ | 3 级风险（normal/warning/critical） |
| 认证保护 | ✅ | admin_auth_middleware |
| 自动轮询 | ✅ | 30s 定时刷新 |
| 搜索/筛选/快捷过滤 | ✅ | 关键字/状态/风险等级 |

## 4. 核心后端设计

### 4.1 资源监控

`managed_project_sites.rs` 使用 `sysinfo` crate 实现系统级监控：

- **CPU/内存/磁盘**: 通过 `ResourceSampler`（`OnceLock<Mutex<...>>`）缓存采样
- **风险阈值**: 明确的常量定义
  - CPU: 85% warning / 95% critical
  - 内存: 80% warning / 90% critical
  - 磁盘: 85% warning / 95% critical
  - 进程 CPU: 70% warning / 90% critical
  - 进程内存: 1.5GB warning / 3GB critical
  - 解析耗时: 10min warning / 30min critical

### 4.2 进程管理

每个站点独立管理 3 个进程：
- SurrealDB 进程 (db_pid)
- web_server 进程 (web_pid)  
- 解析进程 (parse_pid)

通过 `tokio::process::Command` 异步启动，PID 写入 SQLite 跟踪。

### 4.3 日志系统

3 类日志独立文件：
- 解析日志（parse.log）
- 数据库日志（db.log）
- 站点日志（web.log）

通过 `LogSnapshot` 结构提取尾部内容和关键日志行。

## 5. 审核结论

### 5.1 优点

1. **认证完善**: 所有 API 经过 `admin_auth_middleware`
2. **错误处理规范**: `anyhow` 错误链 + `admin_response` 统一响应格式
3. **Store 简洁**: 103 行，职责单一，`withAction` 防重复提交
4. **资源监控完整**: 机器级 + 进程级 + 解析耗时三层风险评估
5. **轮询设计合理**: 30s 间隔，使用 `usePolling` composable
6. **进程生命周期管理**: 完整的 PID 跟踪和清理

### 5.2 发现的问题

| # | 级别 | 问题 | 说明 |
|---|------|------|------|
| 1 | P2 | `managed_project_sites.rs` 过大 | 2600+ 行单文件，建议按职责拆分（CRUD / 进程管理 / 资源监控 / 日志） |
| 2 | P3 | 异地协同缺少认证 | 已知问题，`/api/remote-sync/*` 未经过 `admin_auth_middleware` |
| 3 | P3 | 两套 SQLite 连接逻辑 | `managed_project_sites` 用 `OnceLock`，`remote_sync_handlers` 用 `Once`，建议统一 |

### 5.3 建议：统一异地协同认证

将 `/api/remote-sync/*` 路由也纳入 `admin_auth_middleware` 保护。这是最高价值的改进。
