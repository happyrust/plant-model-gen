## 1. 结论摘要

当前项目里的后台入口已经统一到 **`/admin` 三个页面**：

### 1.1 `/admin`：本机多站点进程编排后台

这是一个 **已经可用** 的独立后台，不走 `web_console` 壳层，入口固定是 `/admin`。

它当前已经支持：

- 新建站点
- 编辑站点
- 删除站点
- 解析项目数据
- 启动站点
- 停止站点
- 查看运行态
- 查看解析 / 数据库 / 站点日志
- 自动刷新
- 本地检索

它的定位不是“注册表展示”，而是 **在当前机器上为某个项目生成独立运行目录、独立 DbOption、独立 SurrealDB 端口、独立 web_server 进程** 的编排工具。

### 1.2 `/api/deployment-sites` / `/admin/#/registry`：中心注册表站点管理

这一套更偏向 **部署注册表管理**，核心能力是：

- 站点列表 / 分页 / 搜索 / 过滤
- 新建 / 编辑 / 删除注册记录
- 从 `DbOption.toml` 导入
- 手动健康检查
- 导出配置
- 创建任务

但它 **不负责** 本机站点进程的解析、启动、停止、日志编排。

### 1.3 `/admin/#/collaboration`：异地协同正式入口

异地协同相关能力已经统一进入 admin 工作台，正式入口为：

- `/admin/#/collaboration`

旧入口：

- `/remote-sync`
- `/console/sync/remote`

现在都只保留兼容跳转。

### 1.4 `/console/deployment/sites`：兼容跳转入口

`/console/deployment/sites` 现在只保留兼容跳转，访问后会直接进入 `/admin/#/registry`。

### 1.5 当前最大问题

当前站点管理已经分成两个明确产品面：

- **`/admin` 已经做了可用的本机编排能力**
- **`/admin/#/registry` 已经承接中心注册表能力**
- **`/admin/#/collaboration` 已经承接异地协同能力**
- **`/deployment-sites` 与 `/console/deployment/sites` 只做兼容跳转**
- **`/remote-sync` 与 `/console/sync/remote` 只做兼容跳转**
- **底层仍是两套数据模型，但入口已经统一**

所以如果问“现在 admin 提供了哪些功能、还缺哪些功能”，答案应该至少按这三个入口来拆：

- `/admin/#/sites`
- `/admin/#/registry`
- `/admin/#/collaboration`

## 3.3 创建站点

### 前端行为

用户点击 `新建站点`，填写表单后点击 `保存站点`：

- 前端校验 `manual_db_nums` 是否为正整数列表
- 发起 `POST /api/admin/sites`
- 成功后提示 `站点已创建`
- 自动刷新列表
- 自动选中新创建站点
- 刷新详情 / 运行态 / 日志面板

### 后端行为

`managed_project_sites::create_site()` 会：

- 校验：
  - 项目名不能为空
  - 项目路径不能为空
  - 项目代号必须大于 0
- 校验端口不与已有站点冲突
- 校验端口不与当前机器已有进程冲突
- 生成 `site_id = slug(project_name) + web_port`
- 在 `runtime/admin_sites/{site_id}` 下生成运行目录
- 生成：
  - `DbOption.toml`
  - `DbOption-parse.toml`
  - `metadata.json`
  - 日志目录
  - 数据目录
- 将记录写入 SQLite 表 `managed_project_sites`

### 创建后默认状态

- `status = Draft`
- `parse_status = Pending`
- `entry_url = http://127.0.0.1:{web_port}`

## 3.4 编辑站点

### 已实现行为

- 从左侧选择站点后，表单进入编辑模式
- 点击保存后发起 `PUT /api/admin/sites/{id}`
- 运行中站点禁止修改
- 修改后会重新生成配置文件并重置运行态

### 后端规则

`managed_project_sites::update_site()` 当前规则：

- 如果站点处于 `Running / Starting / Stopping`，直接拒绝更新
- 允许更新：
  - 项目名
  - 项目路径
  - 项目代号
  - `manual_db_nums`
  - 绑定地址
  - DB 端口
  - WEB 端口
  - 数据库用户名密码
- 更新后会重置：
  - `status = Draft`
  - `parse_status = Pending`
  - `db_pid/web_pid/parse_pid = None`
  - `last_error = None`

### 当前缺陷

前端 `admin.js` 的 `collectUpdatePayload()` **没有把 `manual_db_nums` 带上**。

这意味着：

- 新建时可以设置 `manual_db_nums`
- 编辑时界面上可以看到和修改 `manual_db_nums`
- 但保存更新时，这个字段实际上不会提交给后端

这是当前 `/admin` 最明确、最确定的功能缺口之一。

## 3.5 站点解析流程

### 解析触发方式

点击详情面板中的 `解析` 按钮，调用：

- `POST /api/admin/sites/{id}/parse`

### 解析后端实际流程

`managed_project_sites::parse_site()` 会异步启动 `run_parse_pipeline()`：

1. 如果解析正在运行，则拒绝重复发起
2. 检查数据库是否已启动，未启动则先拉起 SurrealDB
3. 生成解析所需参数
4. 启动 `aios-database` 解析进程
5. 解析结束后回写状态
6. 如果是解析阶段临时拉起的数据库，则解析完成后会停掉数据库

### 解析命令行为

解析进程会根据环境选择：

- 优先直接运行相邻二进制 `aios-database`
- 否则使用 `cargo run --bin aios-database -- -c ...`

如果 `manual_db_nums` 只有一个值，还会追加 `--dbnum`

如果是多个 `manual_db_nums`，会在 `DbOption-parse.toml` 中生成 `included_db_files`

### 解析结果

成功：

- `status = Parsed`
- `parse_status = Parsed`

失败：

- `status = Failed`
- `parse_status = Failed`
- `last_error = 解析失败，退出码: ...`

## 3.6 启动站点

### 启动触发方式

点击 `启动` 按钮，调用：

- `POST /api/admin/sites/{id}/start`

### 启动后端实际流程

`managed_project_sites::start_site()` 会异步启动 `run_start_pipeline()`：

1. 如果站点已经处于 `Running / Starting`，拒绝重复启动
2. 如果站点端口已被占用，拒绝启动
3. 如果解析仍在运行，拒绝启动
4. 标记状态为 `Starting`
5. 保证数据库已启动
6. 如果 `parse_status != Parsed`，先自动执行解析
7. 启动当前 `web_server` 可执行文件
8. 轮询 `http://127.0.0.1:{web_port}/api/status`
9. 若健康检查成功，则状态切换为 `Running`
10. 若启动失败，会回滚并记录 `last_error`

### 当前语义

“启动”不是单纯起 web_server，而是：

- 先保证数据库
- 再保证已解析
- 最后拉起站点

因此它本质上是一个 **组合编排动作**。

## 3.7 停止站点

### 停止触发方式

点击 `停止` 按钮，调用：

- `POST /api/admin/sites/{id}/stop`

### 停止后端实际流程

`managed_project_sites::stop_site()` 会：

- 先将状态置为 `Stopping`
- 尝试按记录中的 `web_pid / db_pid / parse_pid` 杀进程
- 再按端口扫描补杀 `web_port / db_port` 的残留进程
- 最终将状态设置为：
  - `status = Stopped`
  - `db_pid = None`
  - `web_pid = None`
  - `parse_pid = None`
  - `last_error = None`

### 当前特点

这是一个 **强停止** 行为，不是温和的任务编排停止，也没有中间确认步骤。

## 3.8 删除站点

### 删除触发方式

点击 `删除` 按钮后，前端会弹出 `window.confirm`

然后调用：

- `DELETE /api/admin/sites/{id}`

### 删除后端规则

- 站点不存在：返回 `false`
- 站点处于 `Running / Starting / Stopping`：禁止删除
- 可删除时：
  - 删除 SQLite 记录
  - 删除 `runtime/admin_sites/{site_id}` 目录

## 3.9 运行态查看

### 运行态接口

- `GET /api/admin/sites/{id}/runtime`

### 运行态展示内容

前端会展示：

- 站点状态
- 解析状态
- 数据库 PID
- 站点 PID
- 解析 PID
- 数据库端口
- 站点端口
- 入口地址
- 数据库是否运行
- 站点是否运行
- `last_error`

### 运行态判断逻辑

后端不是只信数据库里的 PID，而是会动态检查：

- `kill -0 pid`
- 端口是否可连通

因此页面上的运行态是带一定实时性的，不完全依赖持久化字段。

## 3.10 日志查看

### 日志类型

## 5. `/admin` 还缺哪些功能

下面是基于代码事实整理出来的缺口，按优先级分层。

## 5.1 P0：应该优先补齐

### 5.1.1 编辑站点时 `manual_db_nums` 实际不会保存

这是明确缺陷，不是建议项。

现状：

- 后端支持更新 `manual_db_nums`
- 前端编辑表单也展示了该字段
- 但 `collectUpdatePayload()` 没有把它提交出去

影响：

- 用户会误以为自己已经修改成功
- 实际解析范围不会变化

### 5.1.2 操作按钮没有状态禁用规则

当前 `解析 / 启动 / 停止 / 删除` 按钮只要选中站点就都能点，前端没有根据状态做禁用。

结果：

- 用户可以在 `Starting` 时继续点 `启动`
- 用户可以在 `Running` 时继续点 `解析`
- 用户可以在解析中继续频繁触发操作
- 只能依赖后端返回错误

这会造成交互噪音和重复请求。

### 5.1.3 没有动作中的 loading / 防重复提交

当前按钮点击后只弹 toast，不会：

- 进入 loading
- 禁止重复点击
- 显示当前执行中动作

对于异步编排动作来说，这会让用户对系统状态缺少把握。

### 5.1.4 创建 / 编辑时没有校验项目路径是否存在、是否可访问

后端创建时只校验：

- 项目名非空
- 项目路径非空
- 项目代号 > 0
- 端口可用

但不会在保存阶段校验：

- 路径是否存在
- 是否有读取权限
- `manual_db_nums` 对应的 db 文件能否找到

这些问题会延后到解析或启动阶段才暴露，成本较高。

### 5.1.5 没有鉴权 / 权限控制

`/admin` 当前是一个裸露的管理后台：

- 没有登录
- 没有角色权限
- 没有操作审计

如果该页面对外暴露，这会是高风险点。

## 5.2 P1：强烈建议补齐

### 5.2.1 缺少可视化状态机与动作可用性提示

当前用户只能看到状态字符串，不知道：

- 什么时候应该先解析
- 什么时候可以直接启动
- 什么时候只能停止
- 为什么按钮失败

建议补充：

- 状态说明
- 动作可用性文案
- 最近一次动作结果

### 5.2.2 缺少配置预览 / 导出能力

当前 `/admin` 会生成配置文件，但页面不支持：

- 预览生成的 `DbOption.toml`
- 下载配置
- 对比新旧配置
- 查看解析配置

### 5.2.3 缺少日志下载、日志持续流式刷新

当前日志只有：

- 拉取最后 120 行
- 手动/轮询刷新

缺少：

- 实时流式 tail
- 下载日志文件
- 清理日志
- 日志关键字搜索

### 5.2.4 健康检查已补齐，但还缺历史与批量能力

`/admin/#/registry` 已经提供单站点健康检查按钮，但还没有：

- 健康检查历史
- 批量健康检查
- 健康失败原因摘要

### 5.2.5 缺少批量操作

当前只能逐个站点：

- 解析
- 启动
- 停止
- 删除

没有批量编排能力。

### 5.2.6 `entry_url` 固定为 `127.0.0.1`

当前代码里生成入口地址时基本固定使用：

- `http://127.0.0.1:{web_port}`

这对本机调试友好，但对以下场景不友好：

- 绑定 `0.0.0.0`
- 远端访问
- Nginx 反代
- 指定公网地址

### 5.2.7 `DbOption` 导入已补齐，但仍偏基础

`/admin/#/registry` 已经提供显式“从 DbOption 导入”能力。

当前还缺：

- 导入历史
- 常用路径建议
- 导入前差异预览

## 5.3 P2：中长期优化项

### 5.3.1 入口已统一到 `/admin`，但仍保留兼容旧地址

当前：

- `/admin/#/sites` 是本机编排正式入口

没有展示：

- CPU
- 内存

### 5.3.3 缺少操作审计

没有记录：

### 5.3.4 缺少远端机器编排

## 7. 建议的完整交互文档

下面这部分是建议版的“完整站点管理交互文档”，用于后续产品实现或页面重构。

## 7.1 页面定位

### 页面名称

站点管理 / 站点编排后台

### 页面目标

让管理员能够在一个页面内完成：

- 站点创建与编辑
- 配置导入与导出
- 解析、启动、停止、删除

### 当前统一入口

- `/deployment-sites`

### A. 顶部概览区

## 7.4 动作与交互规则

## 7.4.1 新建站点

### 新建站点用户流程

1. 点击 `新建站点`
2. 填写表单
3. 点击 `保存`
4. 前端先做字段校验
5. 后端创建记录、生成配置和运行目录
6. 成功后：
   - toast 成功
   - 自动选中新站点
   - 展示详情
   - 默认停留在详情页

### 必须补充的校验

- 项目路径存在
- 端口范围合法
- 端口未占用
- `manual_db_nums` 合法
- 如果填写 `manual_db_nums`，应可选提前校验 db 文件存在

## 7.4.2 编辑站点

### 编辑站点用户流程

1. 选择站点
2. 点击 `编辑`
3. 修改字段
4. 点击 `保存`
5. 如果站点正在运行，提示先停止后再编辑
6. 保存成功后：
   - 重置状态为待解析
   - toast 成功
   - 刷新详情

### 必须规则

- `manual_db_nums` 必须可更新
- 若修改了端口，需要重新做端口冲突校验
- 若修改了路径，建议提示“下次启动将重新解析”

## 7.4.3 站点解析

### 站点解析用户流程

1. 选中站点
2. 点击 `解析`
3. 前端弹确认：
   - 是否确认重新解析
   - 是否只解析指定 dbnum
4. 提交后按钮进入 loading
5. 页面进入“解析中”状态
6. 自动轮询运行态和日志
7. 结束后显示成功或失败结果

### 解析结果反馈

- 成功：显示 `解析完成`
- 失败：显示错误摘要，并自动切到日志面板

## 7.4.4 启动站点

### 启动站点用户流程

1. 选中站点
2. 点击 `启动`
3. 若尚未解析，提示“将先自动解析再启动”
4. 确认后进入启动流程
5. 展示：
   - 启动数据库
   - 解析中（如需要）
   - 启动站点中
   - 健康检查中
6. 成功后显示入口地址并允许一键打开

### 启动交互补充建议

- 启动步骤条
- 启动耗时
- 健康检查响应详情

## 7.4.5 停止站点

### 停止站点用户流程

1. 选中站点
2. 点击 `停止`
3. 弹出确认：
   - 将停止站点进程、数据库进程以及解析任务
4. 执行停止
5. 页面刷新为 `Stopped`

### 停止交互补充建议

- 支持“仅停 web / 保留 DB”高级模式
- 支持 graceful stop / force stop 两档

## 7.4.6 删除站点

### 删除站点用户流程

1. 选中站点
2. 点击 `删除`
3. 弹窗明确说明：
   - 会删除站点记录
   - 会删除运行目录
   - 不允许删除运行中站点
4. 二次确认后执行删除
5. 成功后回到列表空状态或选中下一个站点

## 7.4.7 查看日志

### 日志查看建议交互

- 标签页：解析 / 数据库 / 站点
- 支持自动滚动
- 支持下载日志
- 支持关键字过滤
- 支持展开查看完整错误堆栈

## 7.4.8 健康检查

### 健康检查建议交互

- 单站点健康检查按钮
- 返回：
  - 是否成功
  - 响应码
  - 响应耗时
  - 最近检查时间
- 若失败，应提示：
  - web_server 未启动
  - 接口不可达
  - 健康检查地址配置错误

## 7.5 按钮可用性矩阵

建议采用下面的前端禁用规则。

| 场景 | 解析 | 启动 | 停止 | 编辑 | 删除 |
| --- | --- | --- | --- | --- | --- |
| 未选中站点 | 禁用 | 禁用 | 禁用 | 禁用 | 禁用 |
| `Draft + Pending` | 可用 | 可用 | 禁用 | 可用 | 可用 |
| `Parsed + Parsed` | 可用 | 可用 | 禁用 | 可用 | 可用 |
| `Starting` | 禁用 | 禁用 | 可用 | 禁用 | 禁用 |
| `Running` | 禁用 | 禁用 | 可用 | 禁用 | 禁用 |
| `Stopping` | 禁用 | 禁用 | 禁用 | 禁用 | 禁用 |
| `Failed` | 可用 | 可用 | 视是否仍有进程决定 | 可用 | 可用 |
| `parse_status = Running` | 禁用 | 禁用 | 可用 | 禁用 | 禁用 |

## 7.6 反馈规范

每个动作都应提供三层反馈：

### 动作即时反馈

- 按钮 loading
- toast 提示“已提交请求”

### 动作过程反馈

- 状态 badge 变化
- 最近动作文案
- 日志面板自动刷新

### 动作结果反馈

- 成功 toast
- 失败 toast
- 详情面板保留错误摘要

## 7.7 错误处理规范

建议按错误来源拆分：

### 表单错误

- 字段为空
- 端口非法
- `manual_db_nums` 非法
- URL 非法

### 资源错误

- 路径不存在
- 端口占用
- 配置生成失败
- 权限不足

### 编排错误

- 解析失败
- 数据库启动失败
- 站点启动失败
- 健康检查失败

### 展示规范

- 所有错误都要落在：
  - toast
  - 详情错误区
  - 日志面板

## 8. 建议的实现优先级

## 第一阶段：补齐 `/admin` 当前明显缺口

建议先做：

1. 修复编辑时 `manual_db_nums` 不提交
2. 给 `解析 / 启动 / 停止 / 删除` 增加禁用规则
3. 增加按钮 loading / 防重复点击
4. 创建 / 编辑时补充项目路径可用性校验
5. 增加健康检查按钮

## 第二阶段：增强 `/admin` 的可观测与可用性

1. 日志自动滚动、下载、搜索
2. 配置预览 / 导出
3. 错误面板与启动步骤条
4. 更合理的入口地址展示
5. 批量刷新与批量操作

## 第三阶段：统一到 `/console/deployment/sites`

1. 把当前 placeholder 替换为真实页面
2. 决定是：
   - 将 `/admin` 能力整体迁入 console
   - 还是保留 `/admin` 做本机编排，console 做中心注册表
3. 如果保留双体系，必须在 UI 中明确分工，避免用户混淆

## 9. 最终判断

如果只看 `/admin`，它已经不是一个空壳页面，而是一个 **可用但不完善的本机站点编排后台**。

它现在已经覆盖了最关键的执行链路：

- 创建
- 编辑
- 解析
- 启动
- 停止
- 删除
- 运行态
- 日志

但它还没有达到“完整产品级站点管理”的标准，主要缺在：

- 交互约束不够严谨
- 缺少导入 / 健康检查 / 配置预览等配套能力
- 与 `deployment-sites` / `console` 体系没有统一
- 缺少鉴权、审计与更细粒度状态反馈

如果后续要继续做，我建议把 `/admin` 视为 **现有事实源页面**，优先补齐它的交互缺口，再决定是否收敛到 `/console/deployment/sites`。
