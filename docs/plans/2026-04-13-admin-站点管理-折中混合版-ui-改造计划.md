# Admin 站点管理折中混合版 UI 改造 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在不引入“协同组 / 远程站点 / 流向图”新业务模型的前提下，把 `web-server` 仓库里更成熟的“协同后台 + 运维监控”界面组织方式迁移到 `plant-model-gen` 的 `/admin` 站点管理。

**Architecture:** 保留 `plant-model-gen` 现有 `/api/admin/sites`、`/runtime`、`/logs` 作为事实源，只重做 Vue admin 前端的信息架构、状态表达和组件分层。总览页借 `collaboration` 页的后台感，详情页借 `remote-sync/monitor` 页的状态卡与监控区，但不迁移 React/Next.js 代码和协同业务语义。

**Tech Stack:** Vue 3, TypeScript, Vite, Pinia, Vue Router, lucide-vue-next, 现有 Rust `web_server` `/api/admin/*` 接口

---

## 1. 文档定位

### 1.1 目标读者

- 负责实现 `ui/admin` 的前端开发者
- 负责验证 `/admin` 编排后台真实链路的联调人员
- 负责后续统一 `/admin` 与 `/console` 入口的维护者

### 1.2 本计划解决的问题

当前 `/admin` 已经能完成：

- 新建站点
- 编辑站点
- 解析项目
- 启动站点
- 停止站点
- 查看运行态
- 查看三类日志

但页面仍偏“工具页”，缺少以下体验：

- 首页不像工作台，缺少管理后台感
- 详情页虽然有数据，但重点不突出
- “当前正在做什么”表达不够强
- 日志是纯内容展示，摘要感不足
- 空态、错误态、加载态不成体系

本计划的目标是把这些问题一次性整理清楚，并提供按阶段落地的实现顺序。

### 1.3 非目标

以下内容不在本次改造范围内：

- 不新增“协同组”后端模型
- 不新增“远程站点”后端模型
- 不迁移 `React / Next.js / Recharts / ReactFlow`
- 不引入“流向图、趋势图、告警中心”的完整实现
- 不在第一阶段改造 `/console/deployment/sites`
- 不把 `/admin` 改成多机器编排系统

### 2.3 关键约束

1. **数据模型不能变味**
   - target 的核心对象是 `ManagedProjectSite`
   - 不能硬套 source 的 `Environment / Site / CollaborationGroup`

2. **优先前端改造，不先扩后端**
   - 第一阶段尽量只依赖现有 `/api/admin/*`
   - 只有现有字段明显不足时，再增加后端字段

3. **验证方式遵守仓库约定**
   - 不写测试计划
   - 不以 `test` 命令作为验收路径
   - `web_server` 一律以启动服务 + HTTP 请求验证

4. **构建产物要走真实 admin 入口**
   - `ui/admin` 构建后必须写入 `src/web_server/static/admin`
   - 最终验收入口是 `/admin/#/sites`

## 3. 设计原则

### 3.1 设计目标

把当前页面从“功能可用的工具页”提升为“可持续扩展的站点工作台”。

### 3.2 迁移原则

- **迁结构，不迁业务语义**
- **迁状态表达，不迁图表依赖**
- **迁交互层级，不迁 React 代码**
- **保留 target 的本机编排定位**

### 3.3 页面原则

- 总览页负责“看全局 + 快速操作”
- 详情页负责“看当前 + 处理问题”
- 日志区先做“摘要 + 原文”，不先做复杂检索
- 新建/编辑先保留抽屉，不直接改成向导

## 4. 目标界面方案（折中混合版）

## 4.1 总览页 `/sites`

### 页面目标

让管理员在进入页面后的第一屏就看清：

- 当前总共有多少站点
- 哪些站点在运行
- 哪些站点卡在解析 / 启动 / 错误
- 最近应该关注哪类站点
- 现在最常用的动作入口在哪里

### 信息架构

#### A. 顶部工作台头部

内容：

- 页面标题：`站点管理`
- 副标题：突出“本机站点编排后台”语义
- 主操作：`新建站点`
- 次操作：`刷新`
- 状态提示：最近刷新时间 / 当前过滤结果数

参考来源：

- `collaboration/page.tsx` 的页面头部
- `remote-sync/page.tsx` 的刷新 + 主动作布局

#### B. 核心统计卡

建议 4 张卡：

1. 总站点数
2. 运行中
3. 处理中（`Starting / Stopping / parse_status=Running`）
4. 异常（`Failed` 或最近有 `last_error`）

说明：

- 当前 `SiteStats` 只有 `total / running / error / pending_parse`
- 第一阶段可以在前端本地推导“处理中”
- 如果后面发现计算逻辑过散，再考虑补后端汇总字段

参考来源：

- `remote-sync/monitor/metrics-panel.tsx`
- `collaboration/page.tsx` 的 4 张统计卡

#### C. 快速筛选条

保留现有搜索与状态筛选，并新增：

- 快速筛选 chips：`全部` / `运行中` / `处理中` / `异常` / `待解析`
- “仅显示有入口地址”切换项（可选）

说明：

- 这里不引入服务端过滤，先沿用本地过滤
- 工具栏从“普通输入区”升级为“筛选 + 快速聚焦”区

#### D. 站点列表区

保留表格，但增强每一行的信息密度：

- 项目名 + `site_id`
- 站点状态 badge
- 解析状态 badge
- DB/Web 端口
- 入口地址（可点击）
- 最近错误摘要（有则显示一行）
- 快捷操作：解析 / 启动 / 停止 / 详情

说明：

- 不建议第一阶段直接改成纯卡片墙
- 原因：站点管理更适合高密度比较
- 做法：保留表格结构，提升单元格层级

#### E. 空态 / 错误态 / 加载态

- 空态：强调“先创建第一个站点”
- 错误态：展示错误信息 + 重试按钮
- 加载态：使用骨架块，而不是简单文案

## 4.2 详情页 `/sites/:id`

### 页面目标

让管理员在一个页面内完成：

- 识别当前站点是否正常
- 判断卡在哪个阶段
- 直接触发解析 / 启动 / 停止
- 看最近活动与关键日志
- 必要时再看完整配置和原始日志

### 信息架构

#### A. 顶部详情头部

内容：

- 返回按钮
- 站点名
- `site_id`
- 当前状态 badge
- 主动作按钮：`解析` / `启动` / `停止`
- 次动作：`刷新`
- 如果有 `entry_url`，提供 `打开站点`

参考来源：

- `collaboration/[id]/page.tsx` 的顶部动作区

#### B. 第一屏运行态卡片

建议四张卡：

1. 当前阶段
   - `current_stage_label`
   - `current_stage_detail`
2. 数据库
   - `db_running`
   - `db_pid`
   - `db_port`
3. Web 服务
   - `web_running`
   - `web_pid`
   - `web_port`
4. 解析状态
   - `parse_status`
   - `parse_running`
   - `parse_pid`

说明：

- 这是详情页最重要的一屏
- 目的不是展示更多字段，而是让人 3 秒内理解当前状况

#### C. 最近活动区

内容：

- `recent_activity.label`
- `recent_activity.summary`
- `recent_activity.updated_at`
- `last_key_log`
- `last_key_log_source`
- `recent_log_at`

说明：

- 这个区域是折中混合版的核心
- 它把 source 的“监控感”嫁接到 target 已有 `runtime` 字段上

#### D. 日志区

拆成两层：

1. 日志摘要卡片
   - 来自 `streams`
   - 显示 `label / path / updated_at / line_count / last_key_log`

2. 原始日志标签页
   - `parse`
   - `db`
   - `web`

说明：

- 先不做全文搜索
- 先把“看哪类日志最有用”这件事做好

#### E. 配置区

保留原有配置字段展示，但改成结构化分组：

- 项目配置
- 运行端口
- 路径信息
- 创建/更新时间

说明：

- 默认放在详情页下半屏
- 不和运行态抢第一屏位置

## 4.3 新建 / 编辑抽屉

### 目标

保留当前抽屉式交互，但把表单从“字段堆叠”改成“按概念分组”。

### 分组建议

#### 分组 1：项目信息

- `project_name`
- `project_code`
- `project_path`

#### 分组 2：运行配置

- `bind_host`
- `db_port`
- `web_port`

#### 分组 3：解析范围

- `manual_db_nums`

#### 分组 4：数据库凭据

- `db_user`
- `db_password`

### 交互要求

- 编辑与新建继续复用同一抽屉
- 编辑时顶部显示当前状态 badge
- 表单底部保留错误区
- 第一阶段不改成多步骤向导

## 6. API 映射方案

## 6.1 第一阶段只用现有接口

| 页面区域 | 数据来源 | 备注 |
|---|---|---|
| 总览统计 | `GET /api/admin/sites` | 前端本地聚合 |
| 列表区 | `GET /api/admin/sites` | 保持现有加载链路 |
| 详情基础信息 | `GET /api/admin/sites/{id}` | 保持现有 |
| 运行态卡片 | `GET /api/admin/sites/{id}/runtime` | 已满足需要 |
| 最近活动区 | `GET /api/admin/sites/{id}/runtime` | 已有 `recent_activity` / `last_key_log` |
| 日志摘要卡 | `GET /api/admin/sites/{id}/logs` | 已有 `streams` |
| 原始日志标签页 | `GET /api/admin/sites/{id}/logs` | 已有三类日志 |
| 动作按钮 | `POST /parse|start|stop` | 保持现有 |
| 新建编辑 | `POST/PUT /api/admin/sites` | 保持现有 |

## 6.2 第二阶段再评估是否扩接口

只有在以下情况出现时才扩：

1. 前端推导“处理中 / 异常”规则过于复杂
2. 需要服务端直接返回统计汇总
3. 需要单独健康检查动作
4. 需要“复制入口 / 导出配置 / 从 DbOption 导入”能力

### 候选扩展接口（第二阶段）

- `POST /api/admin/sites/{id}/healthcheck`
- `GET /api/admin/sites/{id}/export-config`
- `POST /api/admin/sites/import-dboption`

说明：

- 这些能力来自 `deployment-sites` 体系
- 不应在第一阶段和 UI 重构同时做

## 10. 风险与规避

### 风险 1：把 source 的协同模型硬塞进 target

**风险**：页面看起来更丰富，但语义变乱。  
**规避**：只迁移 UI 组织方式，不迁 `CollaborationGroup / RemoteSite / Flow / Alert` 业务对象。

### 风险 2：列表页做成大卡片墙，密度不足

**风险**：站点多时难比较。  
**规避**：总览页保留高密度表格，只增强单元格层次。

### 风险 3：详情页继续堆字段

**风险**：信息更多，但仍然不好读。  
**规避**：第一屏只留最关键的四张卡和最近活动区。

### 风险 4：前端拆组件过度

**风险**：文件过碎、维护困难。  
**规避**：只提取明确复用或明显提升可读性的组件，不做无意义抽象。

### 风险 5：只在本地 dev 有效

**风险**：构建后 `/admin` 真实入口失效。  
**规避**：每个阶段都跑 `npm run build`，最终必须通过真实 `web_server` 验证。
