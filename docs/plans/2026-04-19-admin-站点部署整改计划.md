# Admin 站点部署整改计划

本计划将 `plant-model-gen` 的 admin 站点部署功能从"单机开发编排"收敛为"默认安全、状态可控、可观测、可回归验证"的运维功能，并补齐与 `plant3d-web` 的 Viewer 联动约定。

## 范围与边界

- 近期实施范围以 `plant-model-gen` 为主，覆盖：
  - admin 站点创建 / 编辑 / 解析 / 启动 / 停止 / 删除
  - 运行态、日志、错误反馈、Viewer 跳转
  - 部署所需配置、进程启动路径、默认安全策略
- 纳入 `plant3d-web` 联动项，但默认以"兼容现有 query 协议"为主，只做必要的小改动。
- 验证方式遵循仓库约定：不写/不跑测试，使用 `web_server` 运行态 + HTTP `POST/GET` + CLI / PowerShell 脚本验证。
- 不在本轮范围：
  - `rs-core` 改造
  - 分布式 agent / 多机调度
  - 完整 secret manager / 加密凭据系统

## 近期实施计划

### 阶段 1：P0 安全默认值收口

#### 目标

先把"默认不安全"的行为收紧，避免 admin 站点部署功能继续以开发态弱配置运行。

#### 改动点

- `src/web_server/managed_project_sites.rs`
  - 将 SurrealDB 默认绑定从 `0.0.0.0` 改为仅本机可访问。
  - 明确区分：
    - Web 入口对外地址
    - DB 监听地址
  - 不再让 DB 绑定策略隐式跟随当前实现中的硬编码。
- `src/web_server/models.rs`
  - 如有必要，补充显式的 DB 绑定配置字段；若本轮坚持最小改动，则先不暴露前端字段，仅后端强制收口为本机绑定。
- `ui/admin/src/components/sites/SiteDrawer.vue`
  - 去掉 `root/root` 默认凭据填充。
  - 改成显式输入或明确提示"将使用后端安全默认值"。
- `src/web_server/managed_project_sites.rs`
  - 创建/更新时增加凭据校验，拒绝空白或明显弱默认值（推荐直接拒绝 `root/root` 组合）。

#### 完成标准

- 新建站点默认不会把 DB 暴露到所有网卡。
- 用户不再在 UI 中无感知地创建出 `root/root` 的站点。
- 错误信息能明确指出"凭据缺失/过弱/绑定策略不允许"。

### 阶段 2：P0 状态机与动作约束加固

#### 目标

把"创建/解析/启动/停止/删除"的后端状态机收严，避免运行中重解析、启动中无法停止、删除时仍有后台任务等问题。

#### 改动点

- `src/web_server/managed_project_sites.rs`
  - 明确定义动作前置条件：
    - `parse`：禁止在 `Running` / `Starting` / `Stopping` 或 `parse_status=Running` 时触发。
    - `start`：禁止重复启动；若解析中则直接拒绝。
    - `stop`：允许在 `Running`、`Starting`、`parse_status=Running` 等"有活动进程"的状态下执行。
    - `delete`：当 `web/db/parse` 任一进程仍活跃时拒绝删除。
  - 对 `refresh_site()` / `update_runtime()` 的状态回写规则做一次收口，避免状态与真实 PID/端口状态脱节。
  - 对失败分支统一落 `last_error`，保证 UI 可见。
- `ui/admin/src/components/sites/site-status.ts`
  - 调整 `isSiteBusy()`、`canStart/canStop/canParse/canDelete` 的前端判定，与后端规则一致。
- `ui/admin/src/components/sites/SiteDataTable.vue`
- `ui/admin/src/components/sites/SiteDetailHeader.vue`
  - 在 `Starting`、解析进行中等状态下显示"停止/取消当前动作"。
- `ui/admin/src/stores/sites.ts`
- `ui/admin/src/views/SiteDetailView.vue`
  - 对 parse/start/stop/delete 的异常做明确提示，避免 `.then(fetchAll)` 式静默失败。

#### 完成标准

- 运行中的站点无法再次触发 parse。
- 启动中或解析中的站点可以被停止。
- 删除动作不会删除仍在解析/运行的站点运行目录。
- 同一状态下，前后端按钮可见性和后端校验一致。

### 阶段 3：P1 部署环境解耦与路径解析稳定化

#### 目标

把当前依赖 `current_dir()`、源码目录、`cargo run` 回退的开发态逻辑，收敛成更稳定的部署运行方式。

#### 改动点

- `src/web_server/managed_project_sites.rs`
  - 重构 `repo_root()` / `current_exe_path()` / `should_run_aios_database_from_source()` / `aios_database_binary()` 的决策顺序。
  - 优先采用"已发布二进制路径"或"当前程序同目录二进制"。
  - 将源码态 `cargo run` 回退限制在明确的 debug/dev 场景，避免部署环境误触发。
  - 在日志中打印最终采用的可执行文件路径与工作目录，便于排错。
- `src/web_server/managed_project_sites.rs`
  - 创建/更新站点时增加 `project_path` 存在性、可访问性校验。
  - 对缺失二进制、缺失目录、启动超时等情况给出更明确错误。
- 如需要，可新增最小配置项或环境变量文档，约定：
  - `aios-database` 二进制定位方式
  - web_server 自身拉起子进程时的查找规则

#### 完成标准

- 部署链路不再默认依赖"必须从 repo 根目录启动"。
- 在无源码工具链的机器上，也能按二进制路径运行。
- 站点创建时就能尽早发现非法 `project_path`。

### 阶段 4：P1 Viewer 联动与 URL 合约整理

#### 目标

消除 admin 端对 Viewer 地址的硬编码，改为使用稳定的 URL 合约与可配置入口。

#### 改动点

- `ui/admin/src/components/sites/SiteDataTable.vue`
- `ui/admin/src/views/SiteDetailView.vue`
  - 把 `http://localhost:3101` 硬编码替换为可配置 Viewer base URL。
  - 保留 `associated_project` 优先、`project_name` 回退的项目选择策略。
  - 统一构造 query：
    - `backendPort` 或 `backend`
    - `output_project`
- `plant-model-gen` admin 配置侧
  - 约定 Viewer base URL 的来源（推荐环境变量或 admin 配置项）。
- `plant3d-web`
  - 当前已确认 `apiBase.ts`、`useModelProjects.ts` 兼容现有 query 协议；默认不做大改。
  - 如联调中发现约定不统一，再最小调整：
    - `src/utils/apiBase.ts`
    - `src/composables/useModelProjects.ts`
    - `src/App.vue`

#### 完成标准

- admin 页面不再写死 `localhost:3101`。
- Viewer 跳转在本机、反代、远程环境下都能通过配置切换。
- `plant3d-web` 能稳定识别 `backendPort/backend + output_project`。

### 阶段 5：P1 可观测性与错误反馈补强

#### 目标

把"失败了但用户看不出来"改成"失败原因明确、日志能定位、页面能提示"。

#### 改动点

- `ui/admin/src/stores/sites.ts`
  - 为 parse/start/stop/delete 增加动作级错误状态。
- `ui/admin/src/views/SitesView.vue`
- `ui/admin/src/views/SiteDetailView.vue`
  - 增加动作失败 banner / toast / 明确文案。
- `src/web_server/admin_handlers.rs`
  - 视需要细化错误码：参数非法、状态冲突、端口冲突、启动失败分别返回更明确的 message。
- `src/web_server/managed_project_sites.rs`
  - 关键分支统一保留 `last_error`，并在日志中打印命令、退出码、端口检测结果。

#### 完成标准

- 用户能在列表页或详情页直接看到失败原因。
- 运维排查可以先看 UI，再看 `logs/parse.log`、`logs/surreal.log`、`logs/web_server.log`。

## 回归验证方案

### 验证原则

- 使用 debug 模式运行 `web_server`。
- 不使用 `cargo test` / 前端测试 / 独立测试框架。
- 通过 admin API 与现有脚本进行真实链路验证。

### 验证载体

- 现有脚本：`scripts/test-admin-deployment.ps1`
- PowerShell / HTTP 接口：
  - `POST /api/admin/auth/login`
  - `POST /api/admin/sites`
  - `POST /api/admin/sites/{id}/parse`
  - `POST /api/admin/sites/{id}/start`
  - `POST /api/admin/sites/{id}/stop`
  - `DELETE /api/admin/sites/{id}`
  - `GET /api/admin/sites/{id}`
  - `GET /api/admin/sites/{id}/runtime`
  - `GET /api/admin/sites/{id}/logs`

### 回归矩阵

1. **新建站点成功** — 合法 `project_path`、合法端口、非弱默认凭据。
2. **新建站点失败（非法路径）** — `project_path` 不存在或不可访问，后端直接拒绝。
3. **新建站点失败（弱凭据/空凭据）** — 校验错误文案明确。
4. **解析成功** — 观察 `parse_status` 从 `Pending -> Running -> Parsed`。
5. **运行中禁止再次解析** — 返回状态冲突错误，UI 给出明确提示。
6. **启动中/解析中允许停止** — `stop` 后 PID 清理，runtime 回到 `Stopped` 或冲突态。
7. **删除受保护** — 解析中/运行中删除应失败；停稳后删除成功。
8. **Viewer 跳转正确** — URL 包含配置化 base、`backendPort/backend`、`output_project`。
9. **日志可追踪** — 启动失败、端口冲突、缺失二进制时能从 runtime + logs 明确定位。

## 计划产出

- 后端：更严格的站点生命周期与部署环境解析逻辑
- 前端：admin 站点按钮约束、错误提示、Viewer 跳转配置化
- 脚本：更新 `scripts/test-admin-deployment.ps1` 以匹配新约束
- 文档：补充部署与联调约定（Viewer URL 合约、部署依赖、验证步骤）

## 长期演进路线

### 路线 A：从本机编排走向受控运维

- 将 parse / db / web 启动从"直接子进程"演进为"受监督进程"
- 接入系统服务或 supervisor，而非完全依赖 admin 进程持有 PID
- 增加重启策略、进程退出原因追踪、审计日志

### 路线 B：凭据与配置治理

- 将 DB 凭据从明文配置文件迁移到更安全的配置通道
- 区分 UI 可见配置与敏感配置
- 后续再评估 secret manager / OS 凭据存储

### 路线 C：多环境 / 多机部署

- 为远程节点引入 agent 或部署执行器
- admin 只做调度与状态汇总，不直接在同进程内起所有服务
- 站点资源、日志、健康检查改成标准化远程协议

## 建议实施顺序

1. 阶段 1：安全默认值收口
2. 阶段 2：状态机与动作约束加固
3. 阶段 3：部署环境解耦
4. 阶段 4：Viewer 联动整理
5. 阶段 5：错误反馈与回归脚本补强
