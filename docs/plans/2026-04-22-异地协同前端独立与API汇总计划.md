# 异地协同 · 前端独立 + API 汇总到 plant-model-gen

> 制定日期：2026-04-22
>
> 前置文档：`design/collaboration-v2/*`（admin 内嵌版 · 推进中）
>
> 配套 diagrams：`design/collab-consolidation/*.html`（4 张）

## 目标

从 `D:\work\plant-code\web-server` 抽离异地协同的**前端 UI**到独立项目 `D:\work\plant-code\plant-collab-monitor`，同时把异地协同的**后端 API** 汇总进 `plant-model-gen`，让 plant-model-gen 成为唯一后端源。

**原则**：**完全移植，先抽离保证能用起来**。不改 UI 技术栈、不裁剪 handler，优先跑通端到端链路。

## 与现有 collaboration-v2 的关系

两条线并存，**不替代**：

| 通道 | 定位 | 入口 | 状态 |
|---|---|---|---|
| admin 内嵌 `/admin/#/collaboration` | 轻量管理入口（4 Tab 壳） | 内嵌在 admin | collaboration-v2 Phase 1–6/8 已完成 |
| **plant-collab-monitor**（本计划） | 专业监控台（11 视图，MQTT 调试、归档浏览、双实时通道） | 独立 SPA | 未开始 |

两者共用 plant-model-gen 作为唯一后端。

## 技术决策（已拍板）

| 项 | 选择 | 理由 |
|---|---|---|
| UI 栈 | **Naive UI + Tailwind + DaisyUI + FontAwesome** | 复用 web-server 的 8 视图 + 7 组件，工作量最小 |
| API 前缀 | `/api/*` 扁平 | plant-model-gen 单一后端，前端无需多 base |
| 实时通道 | SSE 优先 + WebSocket 补充 | SSE 用于事件广播，WS 用于任务细粒度订阅 |
| 数据库访问 | 全走 API | 前端不直连 SQLite / SurrealDB |
| 部署形态 | 独立 SPA（dev 3200 + 生产反代） | 与 plant-model-gen 解耦，可独立版本化 |
| 裁剪权限 | **完全移植，不裁剪** | 先跑通再说 |

## Phase 1 · API 汇入 plant-model-gen（~5h，focal）

### 1.1 迁入 `site_config_handlers.rs`（NEW，~45min）

**步骤**：
1. 复制 `D:\work\plant-code\web-server\src\web_server\site_config_handlers.rs` 到 `D:\work\plant-code\plant-model-gen\src\web_server\` 下
2. 修正路径常量：把 web-server 侧的 `DbOption.toml` 绝对路径换成 plant-model-gen 用的相对路径（或 env）
3. 在 `src/web_server/mod.rs` 注册 7 条路由：
   - `GET  /api/site-config`
   - `GET  /api/site/info`
   - `POST /api/site-config/save`
   - `POST /api/site-config/validate`
   - `POST /api/site-config/reload`
   - `POST /api/site-config/restart`
   - `GET  /api/site-config/server-ip`
4. `cargo check --features web_server` 通过

**验证**：启动 web_server，`curl http://127.0.0.1:9099/api/site-config` 返回 JSON。

### 1.2 迁入 `mqtt_monitor_handlers.rs`（NEW，~1h）

**步骤**：
1. 复制文件
2. 与 `plant-model-gen/src/mqtt_service/` 模块对齐（web-server 可能有自己的 mqtt client，需换成 plant 侧的）
3. 注册 13 条路由：
   - `GET  /api/mqtt/nodes`
   - `GET  /api/mqtt/nodes/{location}`
   - `POST /api/mqtt/nodes/client-unsubscribed`
   - `GET  /api/mqtt/messages`
   - `GET  /api/mqtt/messages/{message_id}`
   - `GET  /api/mqtt/broker/logs`
   - `POST /api/mqtt/subscription/start`
   - `POST /api/mqtt/subscription/stop`
   - `POST /api/mqtt/subscription/clear-master-config`
   - `GET  /api/mqtt/subscription/status`
   - `GET  /api/mqtt/subscription/status/stream`（SSE）
   - `POST /api/mqtt/node/set-master`
   - `POST /api/mqtt/node/set-client`

**验证**：`curl /api/mqtt/nodes` 返回数组。

### 1.3 合并 `sync_control_handlers.rs`（MERGE，~1.5h）

**步骤**：
1. 用 `diff -u plant-model-gen/src/web_server/sync_control_handlers.rs web-server/src/web_server/sync_control_handlers.rs > sync_handlers.diff`
2. 逐项审查 web-server 独有的 handler（估计 ~43KB 增量），按功能分组：
   - 任务管理：`add_sync_task`、`trigger_download`、`cancel_task`、`queue/clear`
   - 运行时：`start`、`stop`、`pause`、`resume`、`restart`
   - MQTT 控制：`mqtt/start`、`mqtt/stop`、`mqtt/status`
3. 把缺失 handler 复制到 plant-model-gen 侧，确保签名、返回类型一致
4. 在 `mod.rs` 补齐缺失路由

**验证**：逐个 endpoint 用 curl 测试。

### 1.4 反向合并 `remote_sync_handlers.rs`（REVERSE MERGE，~1h）

**策略**：plant-model-gen 侧 101KB 为主（功能更全），仅补 web-server 独有的部分。

**步骤**：
1. `diff -u`
2. 标记 web-server 独有 handler，合并进 plant 侧
3. 确认 plant 已有的 20+ 条 `/api/remote-sync/*` 路由都能工作

### 1.5 对齐 `sync_control_center.rs` 和 `increment_manager.rs`（ALIGN，~45min）

**重点**：确保以下核心函数在 plant-model-gen 侧存在且行为一致：
- `SyncControlCenter::start / stop / pause / add_task / get_next_task / complete_task / spawn_worker`
- `AiosDBManager::init_watcher / async_watch / execute_incr_update / poll_sync_e3d_mqtt_events_with_backoff`

**风险**：两侧可能有分叉语义，需要人工审查决定保留哪一版。保守做法：**plant 优先**，仅补 web 独有的功能点。

### 1.6 `deployment_sites.sqlite` schema 对齐（~30min）

**步骤**：
1. 两侧各跑 `sqlite3 deployment_sites.sqlite .schema > schema_web.sql` 和 `schema_plant.sql`
2. `diff` 对比
3. 如有缺失，写 migration SQL 补齐 `remote_sync_envs` / `remote_sync_sites` / `remote_sync_logs`

### 1.7 路由注册总闸（~15min）

确认 `src/web_server/mod.rs` 内所有 `/api/(remote-sync|site-config|mqtt|sync)/*` 路由都已挂载。

## 风险清单

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| `sync_control_center` 两侧分叉（54K vs 44K） | 高 | 高 · 合并语义错位 | Phase 1.5 前跑 `diff -r`，逐 handler 审查，plant 为主 |
| MQTT / SSE broker 连接逻辑 plant 侧偏薄 | 中 | 中 · 实时推送失败 | Phase 1.2 发现困难时，先降级为只迁 site_config + sync，MQTT 监控延后 |
| `DbOption.toml` schema 两侧分叉 | 中 | 中 · site_config 读写失败 | Phase 1.1 预留 schema 兼容层（fallback 字段） |
| 8 Vue 组件 vue-tsc 严格模式报错 | 中 | 低 | 批量跑 `vue-tsc -b`，分批修；可先 `skipLibCheck: true` |
| 实时通道（WS/SSE）前后端协议不一致 | 中 | 中 · 数据断流 | Phase 3 专项验证；如发现不一致，先走轮询 |
| plant-model-gen legacy redirect（`/remote-sync` → `/admin/#/collaboration`）拦截新 API | 低 | 中 | 确认 `mod.rs:610-860` redirect 白名单不匹配 `/api/*` 路径 |
| 两边 `deployment_sites.sqlite` 数据不一致 | 低 | 中 | Phase 1.6 合并前先 dump 数据对比；保留 web-server 数据作为 seed |
| admin collaboration-v2 Phase 7（4-Tab 重构）与本计划并行冲突 | 低 | 低 | 两条线技术栈隔离（admin 用 shadcn-vue 自带组件；monitor 用 Naive UI），代码互不干扰 |
