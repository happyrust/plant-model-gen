# 模型加载缓存优化方案

## 1. 目标
提升大模型加载速度，减少网络带宽消耗，优化首屏体验（实现秒开模型树）。针对移动端和弱网环境尤其重要。

## 2. 缓存分层架构

我们采用四层缓存策略，从最底层的网络请求到最上层的内存实例。

### Layer 1: HTTP 静态资源缓存 (浏览器原生)
*   **对象**: `.glb` 模型文件, `.jpg/.png` 纹理图片。
*   **机制**: 依赖浏览器标准的 HTTP 缓存机制。
*   **策略**: 
    *   配置 Nginx/Web Server 响应头 `Cache-Control: max-age=31536000, immutable`（强缓存）。
    *   **关键**: 文件名必须包含内容的 Hash 或版本号 (Versioning)，例如 `model.v1.0.1.glb` 或 `model.a1b2c3d4.glb`。一旦文件内容改变，URL 必须改变。
    *   对于不带版本的 URL，使用 `ETag` 或 `Last-Modified` 进行协商缓存 (304 Not Modified)。

### Layer 2: 解析后几何数据缓存 (IndexedDB)
*   **对象**: GLB 解析后提取出的纯二进制数据 —— `Float32Array` (Positions, Normals) 和 `Uint32Array` (Indices)。
*   **痛点解决**: 
    *   GLB 下载虽然有 HTTP 缓存，但每次加载仍需 CPU 进行 Decode (JSON Parse) 和 Buffer Extraction (Slice/Copy)。
    *   对于大模型，解析过程可能阻塞主线程。
    *   解析后的 TypedArray 可直接传递给 GPU/Xeokit，速度最快。
*   **存储方案**: IndexedDB
    *   **Store Name**: `geometry_cache`
    *   **Key**: `refno` + `version_hash`
    *   **Value**: 解析好的几何数据对象。

### Layer 3: 元数据与场景树缓存 (LocalStore / IndexedDB)
*   **对象**: `lazyData` (Instance 列表、Transform 矩阵、颜色)、模型树层级结构 JSON。
*   **痛点解决**: 目前必须等待 GLB 下载并解析完毕，才能构建左侧的“模型树”。用户在等待期间看到的是空白。
*   **策略**: 
    *   将模型树结构和构件属性独立存储。
    *   进入页面时，**优先读取缓存**，立即绘制左侧模型树 UI（显示为“离线/加载中”状态）。
    *   后台静默加载 GLB 几何数据，加载完成后点亮 3D 视图。

### Layer 4: 运行时内存缓存 (JS Heap)
*   **对象**: Xeokit `SceneModel` 实例, `geometryCfgById` 映射表。
*   **现状**: 代码中已实现 (`this.geometryCfgById`)。
*   **优化**: 
    *   实现 LRU (Least Recently Used) 清理机制。
    *   监控内存占用，当显存/内存不足时，自动释放不可见的模型资源。

## 3. 详细设计 (Layer 2 - IndexedDB)

### 3.1 数据库 Schema 设计
建议使用 `idb` 库简化 IndexedDB 操作。

```typescript
interface ModelCacheRecord {
  // 联合主键: refno
  id: string;             
  
  // 版本校验字段 (来自服务端 ETag 或 API 返回的 hash)
  version: string;
  
  // LRU 清理用
  lastAccessed: number;
  
  // 核心数据 payload
  data: {
    primitive: 'triangles';
    positions: Float32Array; // 顶点
    normals?: Float32Array;  // 法线
    indices?: Uint32Array;   // 索引
    edgeIndices?: Uint32Array; // 边索引 (如果有)
  };

  // 预计算的包围盒 (避免重复计算 min/max)
  statistics: {
    aabb: { min: [x,y,z], max: [x,y,z] };
    memorySize: number;
  };
}
```

### 3.2 加载流程伪代码

```typescript
async function loadModel(refno: string, expectedVersion: string) {
  // 1. 内存缓存检查
  if (memoryCache.has(refno)) return memoryCache.get(refno);

  // 2. IndexedDB 缓存检查
  const cacheRecord = await db.geometry_cache.get(refno);
  if (cacheRecord && cacheRecord.version === expectedVersion) {
    // 命中缓存：更新访问时间，直接返回
    updateLastAccessed(refno);
    return createGeometryFromBuffers(cacheRecord.data);
  }

  // 3. 缓存未命中：走网络请求
  // 下载并解析 GLB (原有的 performant loader 逻辑)
  const glbBuffer = await fetchGLB(refno); 
  const { positions, normals, indices } = extractBuffers(glbBuffer);

  // 4. 异步写入缓存 (不阻塞渲染)
  saveToIndexedDB(refno, expectedVersion, { positions, normals, indices });

  // 5. 渲染
  return createGeometryFromBuffers({ positions, normals, indices });
}
```

## 4. 实施阶段规划

### Phase 1: 引入 IndexedDB 基础能力
*   引入轻量级 IndexedDB 封装库 (如 `idb-keyval` 或 `localforage`)。
*   创建 `CacheManager` 单例类，管理数据读写。

### Phase 2: 改造 `aios-prepack-bundle-loader`
*   修改 `ensureGeometry` 方法，插入缓存检查逻辑。
*   将当前的 buffer 提取逻辑 (`detect glTF` -> `extract BIN`) 封装为独立的可复用函数，以便在缓存未命中时调用。

### Phase 3: 缓存清理与健壮性
*   实现**存储空间配额管理**：例如限制总缓存大小不超过 500MB。
*   实现**版本失效机制**：服务端数据更新后，前端能识别并覆盖旧缓存。
*   实现**异常降级**：当 IndexedDB 不可用或写入失败时，自动降级为无缓存模式，不影响用户使用。

## 5. 待确认问题
1.  **版本控制来源**：目前通过 `auto_locate_refno` 加载时，URL 中似乎没有包含 hash。我们需要确认如何获取模型的唯一版本标识（ETag? 还是 API 里的 `update_time`?）。推荐后端 API 返回模型元数据时带上 `content_hash`。
2.  **iOS/Safari 限制**：IndexedDB 在 iOS Safar 上有配额限制（通常 500MB - 1GB），且不稳定，需要做好兼容性测试。
