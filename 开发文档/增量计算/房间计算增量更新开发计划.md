# 房间计算增量更新开发计划

## 目标
- 在几何数据变更时，仅重算受影响的房间面板与构件关系
- 保持与现有“粗算 + 细算”流程一致，保证结果可回溯
- 兼顾性能、可维护性与可观测性

## 范围
- 增量更新算法设计
- 调用链与接口改造
- 缓存与一致性策略
- 测试与验证

## 阶段计划

### 阶段 1：现状梳理与需求确认
- 梳理房间计算的入口调用链
- 确认当前增量更新相关结构与字段（如 IncrGeoUpdateLog）
- 明确“面板/构件/删除/几何变更”的事件范围
- 明确输出目标：更新 room_relation 的规则与一致性边界

交付物：现状分析与变更事件清单

### 阶段 2：算法设计
- 设计“受影响面板”集合计算策略

  - 变更对象是构件：通过 AABB 重叠查询找出可能包含的面板
- 区分房间节点/PANE 变更与非房间几何变更的更新方向
  - 房间节点或 PANE 变更、新增：正向更新（通过房间计算 belongs）
  - 非房间 element 变更：反向更新（通过构件反查受影响面板）
- 设计“局部重算”流程
  - 清理旧关系
  - 粗算：AABB 过滤
  - 细算：点包含/投票逻辑
  - 写回新关系
- 设计缓存策略（mesh/AABB/Trimesh 缓存失效或刷新）

## 现有实现对接与扩展点（函数级）

### 现有入口与关键函数
- 增量入口：`src/fast_model/room_model.rs` 的 `update_room_relations_incremental(refnos: &[RefnoEnum])`
- 受影响面板查询：`query_panels_containing_refnos(refnos)`
- 删除旧关系：`delete_room_relations_for_panels(panels)`
- 面板重算链：
  - `process_panel_for_room(...)`
  - `cal_room_refnos_with_options(...)` / `cal_room_refnos(...)`
  - `save_room_relate(panel_refno, within_refnos, room_num)`

### 需要新增或扩展的输入结构
- 统一增量输入为 `IncrGeoUpdateLog`（位于 `src/data_interface/increment_record.rs`）
- 补充“变更分类”到房间更新入口：
  - 几何变更：prim/loop/bran/hanger/basic_cata
  - 删除变更：delete_refnos
  - 面板变更：需要通过元素类型或标签识别（依赖现有房间关键字）
  - 房间节点变更：FRMW/SBFR/PANE（若日志未标记，需通过 noun 查询补齐）

### 入口改造建议
- 新增或扩展函数（建议保留旧入口）：
  - `update_room_relations_with_log(update_log: &IncrGeoUpdateLog) -> IncrementalUpdateResult`
- 旧入口 `update_room_relations_incremental(refnos)` 保留为兼容层，内部转换为 `IncrGeoUpdateLog`

## 变更类型与更新方向（核心规则）

### 正向更新：房间节点 / PANE 变化
- 触发条件：FRMW/SBFR/PANE 的新增、删除或几何变更
- 行为：
  - 重新计算受影响房间的 room_panel_map（房间 → 面板）
  - 对受影响面板执行 `process_panel_for_room`，从房间出发重算 belongs
- 说明：新增 PANE 时必须走正向更新，否则会漏算所属构件

### 反向更新：非房间几何 element 变化
- 触发条件：非 PANE 的几何元素新增/删除/变更
- 行为：
  - 通过变更构件的 AABB 反查相交面板
  - 仅重算这些面板的 belongs

## 增量更新算法（函数级步骤）

### 1) 变更分类与路由
- 输入：`IncrGeoUpdateLog`
- 分类规则：
  - FRMW/SBFR/PANE 变更 → 正向更新
  - 非 PANE 几何变更 → 反向更新

### 2) 生成受影响面板集合
- 正向更新：
  - 根据房间节点重建/查询 room_panel_map
  - 直接得到受影响面板集合
- 反向更新：
  - 使用 AABB 或空间索引查询相交面板
- 推荐实现：
  - 新增 `collect_affected_panels(update_log) -> Vec<(RefnoEnum, String)>`
  - 其中 `String` 为 room_num（沿用 `query_panels_containing_refnos` 返回格式）

### 3) 删除旧关系并重算
- 删除：
  - `delete_room_relations_for_panels(&affected_panels)`
- 重算：
  - 对每个 `panel_refno` 调用 `process_panel_for_room(...)`
  - `process_panel_for_room` 需要可接受 `room_num`

### 4) 删除变更处理
- `delete_refnos` 不一定在房间内，需要仍然触发受影响面板计算
- 删除前建议清理 AABB/mesh 缓存（如果存在）

### 5) 缓存与一致性
- mesh/trimesh 缓存：对变更 refno 相关缓存进行失效
- 如果无法精准失效，建议保守清理或引入版本号

## 调用链集成（建议路径）
- 增量数据进入点：`gen_all_geos_data(...)` 或上层服务
- 在完成几何更新后触发：
  - `update_room_relations_with_log(&incr_updates)`
- 如果 `incr_updates` 为空，不触发

## 测试方案补充（函数级）
- 单测：
  - 输入仅包含 `prim_refnos` 时，确保只更新受影响面板
  - 输入包含 `delete_refnos` 时，确保旧关系被清理
  - 面板变更时，确保自身面板被重算
- 回归：
  - 增量结果与全量 `build_room_relations` 的关系一致性比对

### 指定参考号的单元测试用例（模拟变更）
- 说明：需要真实数据库与网格文件，建议加 `#[ignore]` 手动运行

#### 用例 A：PANE 24381/35271（房间面板变更/新增）
- 目标：验证“正向更新”路径（从房间出发重算 belongs）
- 模拟方式：
  - 将 `24381/35271` 标记为房间节点变更（PANE）
  - 进入 `update_room_relations_with_log` 后应走“正向更新”
- 预期验证：
  - `room_panel_relate` 中包含该 PANE
  - 该 PANE 对应的 `room_relate` 关系被重算（数量变化或存在记录）

#### 用例 B：元件 24383/73968（非房间几何变更）
- 目标：验证“反向更新”路径（通过构件反查受影响面板）
- 模拟方式：
  - 将 `24383/73968` 作为非 PANE 变更对象加入增量日志
  - 调用 `update_room_relations_with_log`，应通过 AABB 查询受影响面板并重算
- 预期验证：
  - `query_panels_containing_refnos` 返回非空面板列表
  - 对应面板的 `room_relate` 关系发生更新（存在/变更）

#### 用例 C（可选）：删除场景
- PANE 删除：视为房间节点变更，触发正向更新并清理其关系
- 元件删除：触发反向更新，相关面板的 `room_relate` 记录被删除或重算后消失

## 风险与对策
- 风险：AABB 缺失或不准确导致遗漏受影响面板
  - 对策：缺失时回退到“全量面板扫描”或标记为重算全部
- 风险：缓存长期累积导致内存上升
  - 对策：Trimesh/PlantMesh 缓存增加清理策略
- 风险：增量日志不完整导致结果不一致
  - 对策：补充变更事件来源与完整性校验
