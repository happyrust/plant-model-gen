# SurrealDB tubi_relate 查询分析总结

## 📋 查询语句

```sql
SELECT DISTINCT id[0] as bran_owner
FROM tubi_relate
WHERE aabb.d != NONE
```

---

## 🎯 核心概念

### 1. tubi_relate 是什么？

**tubi_relate** 是 SurrealDB 中的一个**关系表（Edge Table）**，用于存储 BRAN（管道分支）与 TUBI（管道）之间的关系。

```
BRAN (pe:12345) ──[tubi_relate:[12345, 0]]──> TUBI (pe:67890)
                ──[tubi_relate:[12345, 1]]──> TUBI (pe:67891)
                ──[tubi_relate:[12345, 2]]──> TUBI (pe:67892)
```

---

### 2. 复合 ID 结构

```
tubi_relate:[bran_refno, tubi_index]
            ↑           ↑
            |           └─ TUBI 在该 BRAN 下的索引（从 0 开始）
            └───────────── BRAN 的 refno

示例:
tubi_relate:[12345, 0]  // BRAN 12345 的第 0 个 TUBI
tubi_relate:[12345, 1]  // BRAN 12345 的第 1 个 TUBI
tubi_relate:[12345, 2]  // BRAN 12345 的第 2 个 TUBI
```

**关键点**:
- `id[0]` = BRAN refno
- `id[1]` = TUBI 索引
- 索引保证了 TUBI 的顺序

---

### 3. 查询语句分解

#### 第一部分: `SELECT DISTINCT id[0] as bran_owner`

- **id[0]**: 提取复合 ID 的第一个元素（BRAN refno）
- **DISTINCT**: 去重（一个 BRAN 可能有多个 TUBI）
- **as bran_owner**: 命名结果列

#### 第二部分: `FROM tubi_relate`

- 从 tubi_relate 关系表中查询

#### 第三部分: `WHERE aabb.d != NONE`

- **aabb.d**: 包围盒数据（6 个浮点数：xmin, ymin, zmin, xmax, ymax, zmax）
- **!= NONE**: 过滤掉没有几何体的记录
- 只查询有效的 TUBI

## 🔄 完整查询流程

### 步骤 1: 查询所有有 TUBI 的 BRAN

```sql
SELECT DISTINCT id[0] as bran_owner
FROM tubi_relate
WHERE aabb.d != NONE
```

**返回**: `[12345, 67890, 99999]`

---

### 步骤 2: 对每个 BRAN 查询其所有 TUBI

```sql
SELECT *
FROM tubi_relate:[12345, 0]..[12345, ..]
WHERE aabb.d != NONE
```

**ID Range 语法**:
- `[12345, 0]`: 开始 ID（BRAN 12345，索引 0）
- `[12345, ..]`: 结束 ID（BRAN 12345，任意索引）
- `..` 表示"任意值"

**等价于**:
```sql
WHERE id[0] = 12345 AND id[1] >= 0 AND aabb.d != NONE
```

**返回**: 该 BRAN 的所有 TUBI（按索引顺序）

---

### 步骤 3: 按 BRAN 分组

```rust
let mut bran_tubi_map: BTreeMap<RefnoEnum, Vec<TubiRecord>> = BTreeMap::new();

for bran_refno in bran_owners {
    let tubis = query_tubis_for_bran(bran_refno).await?;
    bran_tubi_map.insert(bran_refno, tubis);
}
```

**使用 BTreeMap 的原因**:
- 保持 BRAN 的顺序
- 便于调试和验证
- 导出时保持一致性

## 💡 为什么这样设计？

### 1. 为什么使用复合 ID？

**优点**:
- ✅ 自然表达 BRAN → TUBI 的层级关系
- ✅ 保证 TUBI 的顺序（通过索引）
- ✅ 高效的范围查询（ID Range）
- ✅ 避免额外的排序字段

**示例**:
```
tubi_relate:[12345, 0]  // 第 0 个 TUBI
tubi_relate:[12345, 1]  // 第 1 个 TUBI
tubi_relate:[12345, 2]  // 第 2 个 TUBI
```

自动按索引排序，无需额外的 `ORDER BY`。

---

### 2. 为什么使用 id[0] 而不是 in？

**对比**:

| 方式 | 语法 | 性能 | 说明 |
|------|------|------|------|
| `id[0]` | `SELECT id[0]` | ⚡ 快 | 直接访问 ID 的第一个元素 |
| `in` | `SELECT in` | 🐌 慢 | 需要解析节点引用 |

**结论**: 使用 `id[0]` 更直接、更高效。

---

### 3. 为什么需要 DISTINCT？

**问题**: 一个 BRAN 可能有多个 TUBI

```
tubi_relate:[12345, 0]  → id[0] = 12345
tubi_relate:[12345, 1]  → id[0] = 12345
tubi_relate:[12345, 2]  → id[0] = 12345
```

**不使用 DISTINCT**: `[12345, 12345, 12345]`  
**使用 DISTINCT**: `[12345]`

**结论**: DISTINCT 去重，只返回唯一的 BRAN 列表。

---

### 4. 为什么过滤 aabb.d != NONE？

**原因**:
1. 有些 TUBI 可能没有生成几何体
2. 有些 TUBI 可能在数据库中但没有有效的空间信息
3. 只导出有实际几何体的 TUBI

**示例**:
```
tubi_relate:[12345, 0] { aabb: { d: [0,0,0,10,10,10] } }  ✅ 包含
tubi_relate:[12345, 1] { aabb: { d: NONE } }              ❌ 排除
tubi_relate:[12345, 2] { aabb: NONE }                     ❌ 排除
```

## 🚀 性能优化

### 1. 使用 ID Range 查询

**❌ 不推荐**:
```sql
SELECT * FROM tubi_relate
WHERE id[0] = 12345 AND aabb.d != NONE
```

**✅ 推荐**:
```sql
SELECT * FROM tubi_relate:[12345, 0]..[12345, ..]
WHERE aabb.d != NONE
```

**原因**:
- ID Range 使用索引，性能更好
- 直接定位到特定 BRAN 的记录
- 减少全表扫描

---

### 2. 分批查询

```rust
const TUBI_QUERY_CHUNK: usize = 256;

for chunk in bran_owners.chunks(TUBI_QUERY_CHUNK) {
    // 处理每批 BRAN
}
```

**原因**:
- 避免 SQL 语句过长
- 减少内存占用
- 更好的错误处理

---

### 3. 只查询需要的字段

```sql
-- ❌ 不推荐
SELECT * FROM tubi_relate:[12345, 0]..[12345, ..]

SELECT
```
