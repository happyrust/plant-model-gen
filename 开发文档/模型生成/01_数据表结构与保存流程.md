# 模型生成数据表结构与保存流程

## 概述

本文档详细说明三维模型生成过程中涉及的所有数据表结构、功能以及数据保存流程。

## 数据表分类

### 1. 普通表 (NORMAL)

#### 1.1 `inst_geo` - 几何单元表

**功能**：存储标准化的几何单元数据（单位 mesh）

**主要字段**：
- `id`: 几何哈希 (geo_hash)，用作主键
- `refno`: 原始构件编号
- `meshed`: 是否已网格化
- `bad`: 是否标记为错误
- `aabb`: 局部坐标系下的包围盒
- `pts`: 关键点集合
- `unit_flag`: 是否为单位 mesh（通过 transform 缩放 vs 顶点缩放）

**保存位置**：`src/fast_model/pdms_inst.rs::save_instance_data_optimize()`

**保存时机**：
- CATE/LOOP/PRIM 模型生成时，每个唯一的几何单元都会被插入
- 使用 `INSERT IGNORE` 避免重复插入（相同的 geo_hash 只存一份）

**特点**：
- 支持几何复用：相同参数的几何体只存储一次
- 通过 `geo_hash` 唯一标识，由几何参数计算得出

---

#### 1.2 `inst_info` - 实例信息表

**功能**：存储每个构件实例的元数据和变换信息

**主要字段**：
- `id`: 实例唯一标识（格式：`{refno}_info`）
- `refno`: 构件编号
- `generic_type`: 构件类型（EQUI, PIPE, VALV 等）
- `world_transform`: 世界坐标系变换矩阵
- `ptset`: 点集数据（压缩格式存储）
- `owner_refno`: 所属构件编号
- `owner_type`: 所属构件类型
- `sesno`: 会话号
- `cata_hash`: CATE 类型哈希（用于分组）
- `visible`: 是否可见

**保存位置**：`src/fast_model/pdms_inst.rs::save_instance_data_optimize()`

**保存时机**：
- 所有 CATE/LOOP/PRIM 构件处理完成后统一保存
- 使用批量 INSERT（CHUNK_SIZE=100）

**关联关系**：
- 通过 `inst_relate` 与 `inst_geo` 建立多对多关系

---

#### 1.3 `tubi_info` - 管道信息表

**功能**：存储管道（Tubing）的规格参数信息

**主要字段**：
- `id`: 组合键（格式：`{cata_hash}_{arrive}_{leave}`）
- `cata_hash`: CATE 类型哈希
- `arrive`: 到达点编号
- `leave`: 离开点编号
- `axis_params`: 轴参数（起点、终点、方向等）

**保存位置**：`src/fast_model/pdms_inst.rs::save_instance_data_optimize()`

**保存时机**：
- CATE 模型生成过程中收集 `arrive/leave` 点信息时
- 自动去重：相同 `(cata_hash, arrive, leave)` 只存一份

**使用场景**：
- 用于 BRAN/HANG Tubing 的几何生成
- 通过 `tubi_relate` 关联到具体的 BRAN/HANG 实例

---

### 2. 关系表 (RELATION)

#### 2.1 `inst_relate` - 实例关系表

**功能**：连接 `pe`（构件）和 `inst_info` 的关系表，记录构件的实例化信息

**关系方向**：`pe` → `inst_info`

**主要字段**：
- `id`: 关系标识（格式：`{refno}_inst`）
- `in`: 指向 `pe`（构件编号）
- `out`: 指向 `inst_info:⟨{refno}_info⟩`
- `world_trans`: 世界变换矩阵（引用 `trans` 表）
- `generic`: 构件类型
- `zone_refno`: 所属 ZONE（通过函数自动计算）
- `spec_value`: 规格值（来自 ZONE.owner.spec_value）
- `dt`: 日期时间（会话日期）
- `has_cata_neg`: 是否包含 CATE 负实体
- `solid`: 是否为实体
- `owner_refno`: 所属构件
- `owner_type`: 所属构件类型

**保存位置**：
- `src/fast_model/pdms_inst.rs::save_instance_data_optimize()`
- CATE: `src/fast_model/cata_model.rs`
- LOOP: `src/fast_model/loop_model.rs`
- PRIM: `src/fast_model/prim_model.rs`

**保存时机**：
- 构件处理完成后立即创建关系
- `replace_exist=true` 时会级联删除旧关系

**索引**：
- `idx_inst_relate_zone_refno`: ZONE 编号索引
- `idx_inst_relate_in`: 输入（pe）索引
- `idx_inst_relate_out`: 输出（inst_info）索引

---

#### 2.2 `geo_relate` - 几何关系表

**功能**：连接 `inst_info` 和 `inst_geo` 的关系表，记录实例包含的几何单元

**关系方向**：`inst_info` → `inst_geo`

**主要字段**：
- `id`: 关系标识（哈希值）
- `in`: 指向 `inst_info:⟨{refno}_info⟩`
- `out`: 指向 `inst_geo:⟨{geo_hash}⟩`
- `trans`: 局部变换矩阵（引用 `trans` 表）
- `geom_refno`: 原始几何构件编号
- `pts`: 关键点列表（引用 `vec3` 表）
- `geo_type`: 几何类型（Pos, Neg, Compound, CataNeg, CataCrossNeg）
- `visible`: 是否可见
- `cata_neg`: CATE 负实体列表（可选）

**保存位置**：`src/fast_model/pdms_inst.rs::save_instance_data_optimize()`

**保存时机**：
- 与 `inst_geo` 同时保存（同一批处理）

**特殊用途**：
- `geo_type=Neg`: 用于 `neg_relate` 关系构建
- `geo_type=CataCrossNeg`: 用于 `ngmr_relate` 关系构建

---

#### 2.3 `inst_relate_aabb` - 实例包围盒关系表

**功能**：存储构件的世界坐标系包围盒（AABB）

**关系方向**：`pe` → `aabb`

**主要字段**：
- `id`: 关系标识（格式：`{refno}_aabb`）
- `in`: 指向 `pe`（构件编号）
- `out`: 指向 `aabb:⟨{aabb_hash}⟩`（包围盒记录）

**保存位置**：
- 初始保存：`src/fast_model/pdms_inst.rs::save_instance_data_optimize()`
- 更新：`src/fast_model/mesh_generate.rs::update_inst_relate_aabbs_by_refnos()`

**保存时机**：
- 模型生成阶段：与 `inst_relate` 同时创建
- 网格生成后：根据实际 mesh 顶点重新计算并更新

**注意事项**：
- 必须在 `aabb` 表记录写入**之后**才写入关系，避免空引用

---

#### 2.4 `inst_relate_bool` - 实例布尔运算关系表

**功能**：记录构件布尔运算的结果（孔洞、开槽等）

**关系方向**：`inst_geo` → `inst_geo`（运算结果）

**主要字段**：
- `in`: 原始几何（`inst_geo`）
- `out`: 布尔运算后的几何（`inst_geo`）
- `bool_type`: 布尔运算类型
- `status`: 运算状态（Success, Failed 等）

**保存位置**：`src/fast_model/manifold_bool.rs`（布尔运算模块）

**保存时机**：
- 布尔运算完成后立即创建关系
- 导出时通过 `enable_holes=true` 查询此关系获取最终 mesh

---

#### 2.5 `inst_relate_cata_bool` - CATE 布尔运算关系表

**功能**：记录 CATE 类型构件的复合布尔运算结果

**关系方向**：`inst_info` → `inst_geo`（复合结果）

**主要字段**：
- `in`: CATE 实例（`inst_info`）
- `out`: 布尔运算后的几何（`inst_geo`）
- `operation`: 运算操作类型

**保存位置**：`src/fast_model/cata_model.rs`（CATE 处理模块）

**保存时机**：
- CATE 的多个几何单元进行布尔运算后
- 用于优化渲染性能（减少 draw call）

---

#### 2.6 `neg_relate` - 负实体关系表

**功能**：记录负实体（Neg）对正实体的切割关系

**关系方向**：`geo_relate` → `pe`（正实体）

**主要字段**：
- `id`: 关系标识（格式：`[geo_relate_id, target_pe]`）
- `in`: 指向 `geo_relate:⟨{geo_relate_id}⟩`（切割几何）
- `out`: 指向 `pe`（被切割的正实体）
- `pe`: 负载体（负实体构件编号）

**保存位置**：`src/fast_model/pdms_inst.rs::save_instance_data_optimize()`

**保存时机**：
- 处理 CATE 负实体时自动创建
- 从 `geo_relate` 中筛选 `geo_type=Neg` 的记录

**数据结构演变**：
- **旧结构**：`in=负实体refno, out=正实体refno`
- **新结构**：`in=geo_relate ID, out=正实体refno, pe=负载体refno`

---

#### 2.7 `ngmr_relate` - NGMR 负实体关系表

**功能**：记录 NGMR（非几何负实体）的切割关系

**关系方向**：`geo_relate` → `pe`（正实体）

**主要字段**：
- `id`: 关系标识（格式：`[geo_relate_id, target_pe]`）
- `in`: 指向 `geo_relate:⟨{geo_relate_id}⟩`（CataCrossNeg 切割几何）
- `out`: 指向 `pe`（目标正实体）
- `pe`: 负载体（NGMR 所有者）
- `ngmr`: NGMR 几何引用（用于调试）

**保存位置**：`src/fast_model/pdms_inst.rs::save_instance_data_optimize()`

**保存时机**：
- CATE 处理时检测到 NGMR 关系时创建
- 从 `geo_relate` 中筛选 `geo_type=CataCrossNeg` 的记录

**使用场景**：
- 导出模型时识别哪些几何被 NGMR 切割
- 用于碰撞检测和空间分析

---

#### 2.8 `tubi_relate` - 管道关系表

**功能**：记录 BRAN/HANG 构件的管道（Tubing）实例

**关系方向**：`pe`（管件）→ `pe`（管件）

**重要说明**：
- **Tubing 使用 `tubi_relate` 表，不创建 `inst_relate`**
- `inst_relate` 仅用于管件（TEE、ELBO等）指向 `inst_info:⟨cata_hash⟩`
- 避免 Tubing 覆盖管件的 `inst_relate` 数据

**主要字段**：
- `id`: 关系标识（格式：`[pe:{first_child_refno}, index]`）
- `in`: BRAN/HANG 的第一个子元素 refno
- `out`: 管道实例信息（`inst_info` 或 `tubi_info`）
- `aabb`: 世界坐标系包围盒
- `world_trans`: 世界变换矩阵
- `geo`: 几何哈希（`inst_geo`）
- `date`: 日期时间
- `spec_value`: 规格值

**保存位置**：
- CATE 阶段：`src/fast_model/cata_model.rs`（收集 tubi_info）
- BRAN/HANG 阶段：`src/fast_model/mesh_generate.rs`（生成 tubi_relate）

**保存时机**：
- 先收集 `tubi_info`（CATE 阶段）
- 后创建 `tubi_relate`（BRAN/HANG 几何生成后）

**查询方式**：
- 按 BRAN refno：`tubi_relate:[{pe_key}, 0]..[{pe_key}, 999999]`
- 按流程构件：`WHERE in={pe_key} OR out={pe_key}`

---

## 数据保存流程

### 整体流程概览

```
1. 模型生成阶段（gen_all_geos_data）
   ├─ CATE 处理 → 生成 inst_info, inst_geo, geo_relate
   ├─ LOOP 处理 → 生成 inst_info, inst_geo, geo_relate
   ├─ PRIM 处理 → 生成 inst_info, inst_geo, geo_relate
   └─ 收集 tubi_info

2. 数据保存阶段（save_instance_data_optimize）
   ├─ 批量保存 inst_geo
   ├─ 批量保存 geo_relate
   ├─ 批量保存 inst_info
   ├─ 批量保存 inst_relate
   ├─ 批量保存 inst_relate_aabb
   ├─ 批量保存 tubi_info
   ├─ 批量保存 neg_relate（如果有）
   └─ 批量保存 ngmr_relate（如果有）

3. 网格生成阶段（mesh_generate）
   └─ 更新 inst_relate_aabb（根据实际 mesh 顶点）

4. 布尔运算阶段（manifold_bool）
   └─ 创建 inst_relate_bool 关系
```

### 详细保存流程

#### 阶段 1：模型生成（`gen_all_geos_data`）

**文件位置**：`src/fast_model/gen_model/orchestrator.rs`

1. **CATE 处理**（`src/fast_model/cata_model.rs`）
   - 遍历 BRAN/HANG 分组
   - 生成每个构件的几何单元（`EleInstGeo`）
   - 收集到 `ShapeInstancesData`
   - 收集 `tubi_info`（如果有 arrive/leave 点）

2. **LOOP 处理**（`src/fast_model/loop_model.rs`）
   - 处理 LOOP 类型构件
   - 生成几何单元并收集

3. **PRIM 处理**（`src/fast_model/prim_model.rs`）
   - 处理 PRIM 类型构件
   - 生成几何单元并收集

**数据结构**：`ShapeInstancesData`
```rust
pub struct ShapeInstancesData {
    pub inst_info_map: DashMap<RefnoEnum, EleGeosInfo>,    // → inst_info
    pub inst_geos_map: DashMap<String, EleInstGeosData>,   // → inst_geo + geo_relate
    pub inst_tubi_map: DashMap<RefnoEnum, EleGeosInfo>,    // → tubi_info
    pub neg_relate_map: HashMap<RefnoEnum, Vec<RefnoEnum>>, // → neg_relate
    pub ngmr_neg_relate_map: HashMap<RefnoEnum, Vec<(RefnoEnum, RefnoEnum)>>, // → ngmr_relate
}
```

#### 阶段 2：数据保存（`save_instance_data_optimize`）

**文件位置**：`src/fast_model/pdms_inst.rs`

**批处理机制**：
- `CHUNK_SIZE = 100`：单条 INSERT 的记录数
- `MAX_TX_STATEMENTS = 5`：单事务最大语句数
- `MAX_CONCURRENT_TX = 5`：最大并发事务数

**保存顺序**：

1. **inst_geo + geo_relate**（同时处理）
   ```rust
   for inst_geo_data in inst_mgr.inst_geos_map.values() {
       for inst in &inst_geo_data.insts {
           // 生成 inst_geo JSON
           // 生成 geo_relate JSON（连接 inst_info 和 inst_geo）
           // 收集 Neg/CataCrossNeg geo_relate 用于后续关系构建
       }
   }
   ```

2. **neg_relate**（如果有负实体）
   - 从 `geo_relate` 中查找 `geo_type=Neg` 的记录
   - 构建 `geo_relate → pe` 的关系

3. **ngmr_relate**（如果有 NGMR）
   - 从 `geo_relate` 中查找 `geo_type=CataCrossNeg` 的记录
   - 构建 `geo_relate → pe` 的关系

4. **inst_info + inst_relate + inst_relate_aabb**
   ```rust
   for (key, info) in &inst_mgr.inst_info_map {
       // 1. 生成 inst_info JSON
       // 2. 生成 inst_relate JSON（pe → inst_info）
       // 3. 生成 inst_relate_aabb JSON（pe → aabb）
   }
   ```

5. **tubi_info + tubi_relate**
   - 从 `inst_tubi_map` 收集 transform 和 aabb 数据
   - **注意**：`inst_tubi_map` 不再创建 `inst_relate`（避免覆盖管件数据）
   - Tubing 的关系通过 `tubi_relate` 表单独处理

6. **transform + aabb + vec3 表**
   - 自动去重：相同的变换/包围盒/点只存一份
   - 通过哈希值引用

#### 阶段 3：网格生成后更新（`update_inst_relate_aabbs_by_refnos`）

**文件位置**：`src/fast_model/mesh_generate.rs`

**流程**：
1. 查询已生成的 mesh 顶点
2. 重新计算 AABB（基于实际顶点）
3. 更新 `inst_relate_aabb` 关系

**优化**：
- 批量更新（按 CHUNK_SIZE 分组）
- 使用 `DELETE + INSERT RELATION` 实现幂等更新

## 表间关系图

```
pe (构件表)
  ├─[inst_relate]─→ inst_info (实例信息)
  │                    ├─[geo_relate]─→ inst_geo (几何单元)
  │                    │                 ├─[inst_relate_bool]─→ inst_geo (布尔结果)
  │                    │                 └─[geo_relate:Neg]─→[neg_relate]─→ pe (被切割实体)
  │                    └─[inst_relate_cata_bool]─→ inst_geo (CATE 布尔结果)
  │
  ├─[inst_relate_aabb]─→ aabb (包围盒)
  │
  └─[tubi_relate]─→ inst_info / tubi_info (管道实例)
                        └─[geo_relate]─→ inst_geo (管道几何)

geo_relate
  ├─[neg_relate]─→ pe (负实体切割关系)
  └─[ngmr_relate]─→ pe (NGMR 切割关系)
```

---

## 关键设计原则

1. **几何复用**：相同参数的 `inst_geo` 只存一份，通过 `geo_relate` 多对多关联
2. **变换复用**：相同的 `transform/aabb/vec3` 只存一份，通过哈希引用
3. **关系表优先**：使用 RELATION 类型表存储关系，便于图查询
4. **批量处理**：所有 INSERT 操作都使用批量模式（CHUNK_SIZE=100）
5. **事务批处理**：使用 `TransactionBatcher` 控制事务大小，避免超时
6. **幂等更新**：`replace_exist=true` 时先删除旧数据再插入新数据

## 查询示例

### 查询构件的所有几何实例
```sql
SELECT * FROM geo_relate WHERE in = inst_info:⟨{refno}_info⟩;
```

### 查询负实体切割关系
```sql
SELECT * FROM neg_relate WHERE out = pe:{target_refno};
```

### 查询 BRAN 的管道实例
```sql
SELECT * FROM tubi_relate:[{bran_first_child_refno}, 0]..[{bran_first_child_refno}, 999999];
```

### 查询构件的世界变换和包围盒
```sql
SELECT 
    ir.world_trans,
    ira.out as aabb
FROM inst_relate ir
LEFT JOIN inst_relate_aabb ira ON ir.in = ira.in
WHERE ir.in = pe:{refno};
```

## 查询性能：全表扫描现状与改造建议

本节总结当前代码路径中已定位的“全表扫描 / 等价全量扫描”点，并给出与“遍历 `pe` 必须以 `pe_{dbnum}` 为驱动表”的约束一致的改造方向。

### 判定标准（简述）

1. 查询语句无 `WHERE`/`LIMIT`/`START` 等边界条件，直接对大表做 `SELECT`。
2. 对大表执行 `count()`（尤其 `GROUP ALL`）用于统计展示。
3. 使用 `ORDER BY + LIMIT + START` 循环翻页直到空页（对目标表做“快照式全量遍历”）。
4. 在 TreeIndex/内存索引上遍历 `all_refnos()` 再做过滤（不查 SurrealDB，但等价全量扫描）。

### 已定位点（按风险）

1. **高风险：直接遍历 `pe` 全表**
   - `gen_model-dev/src/dblist_parser/db_loader.rs::get_all_refnos`
     - `SELECT VALUE id FROM pe;`

2. **中高风险：全库统计直接打 `pe`（Web/看板侧）**
   - `pdms-io-fork/surrealdb.ts`
     - `SELECT count() AS total FROM pe`

3. **中风险：对 `inst_geo` 的全量/近似全量遍历（replace 模式）**
   - `gen_model-dev/src/fast_model/mesh_generate.rs::snapshot_mesh_geo_ids_for_replace`
     - `SELECT value id FROM inst_geo ... ORDER BY id LIMIT ... START ...` 循环至空页
   - `gen_model-dev/src/fast_model/mesh_generate.rs::query_total_pending_mesh_count`
     - `SELECT VALUE count() FROM inst_geo ... GROUP ALL`

4. **高风险：Full Noun / TreeIndex 等价全量扫描**
   - `gen_model-dev/src/fast_model/gen_model/tree_index_manager.rs::query_noun_refnos`
     - 对每个 `dbnum` 遍历 `index.all_refnos()` 再按 noun hash 过滤
   - `gen_model-dev/src/fast_model/query_provider.rs::query_by_noun_all_db`
     - `query_noun_refnos(noun, None)`（无上限）
   - `gen_model-dev/src/fast_model/gen_model/full_noun_mode.rs::prequery_noun_counts`
     - 为“计数/调度”先取每个 noun 的全量 refno 列表

### 与 `pe_{dbnum}` 驱动遍历一致的改造约束

1. **遍历约束**：凡“遍历/分页列举 pe”的入口，必须只扫 `pe_{dbnum}`，先取 `id` 再批量组合查询其它表。
2. **统计约束**：Web/看板类“全库总数/按天统计”不得直接 `count() FROM pe`，应使用汇总表/缓存表（如 `dbnum_info_table` 或等价聚合结果）。
3. **Full Noun 约束**：`full_noun_mode` 视作离线/受控模式，必须明确限制：
   - dbnum 范围（如 `manual_db_nums` 或等价过滤）
   - noun 范围（如 `full_noun_enabled_categories`）
   - 单 noun 的 refno 上限（如 `debug_limit_per_noun`）
   - 避免“为计数而全量拉取 refno 列表”的实现方式，优先改为索引化计数/缓存

---

## 注意事项

1. **依赖顺序**：`inst_relate_aabb` 必须在 `aabb` 表记录写入后创建，避免空引用
2. **级联删除**：`replace_exist=true` 时会级联删除 `inst_relate` 及其相关关系
3. **NaN 检查**：插入前会检查 `transform` 是否包含 NaN，跳过无效数据
4. **压缩存储**：`ptset` 使用压缩格式存储，减少约 70-80% 空间
5. **并发控制**：使用 `TransactionBatcher` 限制并发事务数，避免 SurrealDB 连接超限
