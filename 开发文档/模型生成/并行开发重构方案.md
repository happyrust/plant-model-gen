# PRIM/LOOP 页内预取优化完整开发方案（Worktree 执行版）

## 摘要
目标是在不重启旧 `geom_input_cache` 全局两阶段的前提下，将 `PRIM/LOOP` 从“逐 refno 多次查询”改为“按 page 批量预取 + 并发生成消费内存上下文”，并保证失败可降级、不丢数。  
主验收指标：`gen_model` 总耗时下降 `>=20%`。  
已定策略：页内批取、失败降级逐条查询、先 `PRIM` 后 `LOOP`、在 worktree 中实施。

## 范围与非范围
1. 范围
- `src/fast_model/gen_model/prim_model.rs`
- `src/fast_model/gen_model/loop_model.rs`
- 复用查询能力：`src/fast_model/gen_model/query_provider.rs`、`src/fast_model/gen_model/neg_query.rs`、`src/fast_model/gen_model/transform_cache.rs`
2. 非范围
- 不恢复 `src/fast_model/model_cache/geom_input_cache.rs` 旧链路
- 不改 `CATE/BOOLEAN` 逻辑
- 不改外部 CLI 协议与数据库 schema

## Worktree 方案（先执行）
1. 新建隔离分支与 worktree
- 分支名：`feat/prim-loop-page-prefetch`
- worktree 目录：`../gen_model-dev-wt-prim-loop-prefetch`
2. 在 worktree 内完成全部改造、测试、性能对比
3. 主工作区仅用于审阅，不直接改代码

## 关键设计决策（已锁定）
1. 并发模型仍用 Tokio，不引入 Rayon 到生成主链路。
2. 每个 page 先批量预取，再并发生成；生成阶段优先读内存上下文。
3. 预取失败仅影响当前 page：自动回退旧逐条路径继续执行。
4. 发送通道在 `PRIM/LOOP` 局部改为 `send_async`，保障背压时不阻塞 worker 且不丢批。
5. 严格遵守 `ref0 != dbnum`，所有 dbnum 解析只走 `db_meta().get_dbnum_by_refno(...)` 语义路径。

## 接口与类型变更（公共/外部）
1. 外部 API 变更
- 无（函数签名对调用方保持不变）
2. 内部新增类型（文件内私有）
- `PrimPagePrefetchContext`
- `LoopPagePrefetchContext`
3. 可选运行开关（内部）
- 新增环境变量：`AIOS_PRIM_LOOP_PAGE_PREFETCH`，默认开启；设置 `0/false` 时强制走旧路径

## 代码改造清单（决策完整）
1. `src/fast_model/gen_model/prim_model.rs`
- 新增 `prefetch_prim_page(refnos: &[RefnoEnum], db_option: Arc<DbOptionExt>) -> Result<PrimPagePrefetchContext>`
- 批量取数顺序固定：
- `get_attmaps_batch(refnos)`
- `get_world_transforms_cache_first_batch(Some(db_option.as_ref()), refnos)`
- `neg_query::query_descendants_map_by_dbnum(tree_dir, refnos, &GENRAL_NEG_NOUN_NAMES, false)`
- 将 `gen_prim_geos` 的 page 循环拆为：
- `prefetch` 阶段
- `consume` 阶段（tokio 并发）
- `consume` 中优先读 context；缺失项进入 `load_prim_single_fallback(refno)` 旧逻辑
- `build_polyhedron_from_db` 涉及的 POHE/POLYHE 保持原算法；仅把可批取部分前移（属性/transform/neg）

2. `src/fast_model/gen_model/loop_model.rs`
- 新增 `prefetch_loop_page(...) -> Result<LoopPagePrefetchContext>`
- 批量取数顺序固定：
- `get_attmaps_batch(loop_refnos)`
- `get_world_transforms_cache_first_batch(...)`
- `neg_query::query_descendants_map_by_dbnum(..., &GENRAL_NEG_NOUN_NAMES, false)` 作为直接 NEG
- `CMPF` 两段批查：
- 先查每个 root 的 `CMPF` 列表（root->cmpf_map）
- 再对 cmpf roots 批查 NEG，回填 `cmpf_neg_map`
- `gen_loop_geos` 中替换逐条 `get_named_attmap`/NEG 查询路径为 context-first + fallback

3. 发送与背压（仅 PRIM/LOOP）
- 将 `.send(...)` 改为 `.send_async(...).await`
- 统一错误处理：通道关闭直接返回 `Err`，不 `expect` panic

4. 指标与日志
- 新增阶段计数：
- `prefetch_pages_total`
- `prefetch_pages_success`
- `prefetch_pages_fallback`
- `prefetch_att_ms / prefetch_trans_ms / prefetch_neg_ms`
- 末尾打印 PRIM/LOOP 汇总，格式固定，便于对比脚本抓取

## 失败模式与处理
1. 批量属性查询失败：当前 page fallback 逐条查询
2. 批量 transform 失败：当前 page fallback 逐条 `get_world_transform_cache_first`
3. 批量 NEG 查询失败：当前 page fallback 原 `query_provider` 单条路径
4. 通道发送失败：立即返回错误终止当前任务
5. 任一 fallback 都必须记录原因计数，不允许静默

## 测试与验收
1. 单元测试
- `prefetch context` 组装正确性（空输入、部分缺失、全量命中）
- `fallback` 触发后结果完整性（数量与关键字段不缺）
2. 集成测试
- 复用现有 `gen_model` 测试入口，分别跑 PRIM/LOOP 典型样本
- 对比优化前后：实例总数、`neg_relate` 数量一致
3. 性能测试协议
- 同一数据集跑 3 轮，取中位数
- 记录：总耗时、PRIM 耗时、LOOP 耗时、fallback 页数
- 验收门槛：总耗时下降 `>=20%`，且 fallback 页占比低于 `5%`（异常场景除外）
