# 全量模型碰撞检查开发方案

## 1. 目标与背景

利用现有的 **SQLite RTree** 空间索引（粗筛）和 **Parry3D** 几何精算能力（精算），实现对全站模型或指定区域模型的高效碰撞检查。

目标是提供一个快速的管道，能够处理数十万构件级别的碰撞检测，输出精确的碰撞对及相关信息（如穿透深度）。

## 2. 核心技术架构

采用经典的 **Broad Phase (粗筛)** -> **Narrow Phase (精算)** 两阶段架构。

*   **Broad Phase**: 利用 SQLite 虚拟表 `aabb_index` (RTree) 进行基于 AABB (Axis-Aligned Bounding Box) 的相交查询。此阶段用于快速剔除绝大多数不可能碰撞的物体。
*   **Narrow Phase**: 利用 `parry3d` 库加载几何网格 (TriMesh)，进行精确的几何相交测试 (Intersection Test) 或接触测试 (Contact Test)。

## 3. 开发方案详情

### 3.1 阶段一：数据准备与索引构建

现有系统已经维护了 `aabb_cache`，其中包含 `aabb_index` 虚拟表。
*   **前提**: 确保所有参与检测的模型都已生成 AABB 并存入 SQLite。
*   **优化**: 如果需要全量检测，建议使用快照或只读副本以避免对实时生成的锁竞争。

### 3.2 阶段二：粗筛 (Broad Phase)

**策略**: 批量查询或全表自连接 (Self-Join)。

针对全量检查，不建议逐个构件查询 (N 次查询)，建议采用以下两种策略之一：

1.  **分块/空间分区查询**: 将空间划分为大格子（如 50m x 50m x 50m），每次加载一个区域内的所有 ID，然后在内存中进行配对。
2.  **RTree 遍历**: 利用 RTree 的特性，直接获取重叠对。SQLite RTree 模块支持直接查询重叠矩形。
    *   SQL 示例:
        ```sql
        SELECT t1.id, t2.id 
        FROM aabb_index AS t1, aabb_index AS t2 
        WHERE t1.id < t2.id  -- 避免重复 (A-B 和 B-A) 和自交 (A-A)
          AND t1.min_x <= t2.max_x AND t1.max_x >= t2.min_x
          AND t1.min_y <= t2.max_y AND t1.max_y >= t2.min_y
          AND t1.min_z <= t2.max_z AND t1.max_z >= t2.min_z;
        ```
    *   *注意*: 对于数据量极大(10w+)的情况，全表 Join 可能较慢，建议按 `System` 或 `Area` 分批次进行。

### 3.3 阶段三：精算 (Fine Calculation / Narrow Phase)

**算法**: `parry3d` 几何检测

对于粗筛出的每一对 `(id_a, id_b)`:

1.  **加载几何**:
    *   优先使用 **L0 LOD** Mesh (低精度网格) 以提高速度。`src/fast_model/room_model.rs` 中已有 `load_geometry_with_enhanced_cache` 实现。
    *   对于极高精度需求，可配置为加载 L1/L2。

2.  **几何构建**:
    *   构建 `parry3d::shape::TriMesh`。
    *   应用各自的 `World Transform`。

3.  **相交检测**:
    *   使用 `parry3d::query::contact` 计算接触信息。
    *   **判断标准**: `contact.dist <= tolerance` (容差，如 0.001m) 且 `contact.penetration_depth > 0`。

    ```rust
    // 伪代码
    let contact = query::contact(
        &pos_a, &*mesh_a,
        &pos_b, &*mesh_b,
        prediction // 预测距离，通常设为 0 或很小的值
    )?;
    
    if contact.is_some() {
        // 发生碰撞
    }
    ```

### 3.4 并发与性能优化

*   **并发执行**: 使用 `tokio` + `rayon` 或 `futures::stream::buffer_unordered` 并发处理精算任务。
*   **AABB 预过滤**: 在精算前，再次检查两个 Objectspace AABB 在 Worldspace 的精确重叠（Parry3D 的 AABB check 比 Mesh check 快得多），作为第二道防线。
*   **缓存**: 几何数据 (TriMesh) 应在内存中缓存 (LRU Cache)，因为一个构件可能与多个构件发生碰撞检测。现有 `aabb_cache.rs` 中的缓存机制复用。

## 4. 输出数据结构

建议定义如下结构用于存储结果：

```rust
struct CollisionEvent {
    pair: (RefU64, RefU64), // 碰撞对 (保证顺序，如 id1 < id2)
    location: Point3<f32>,  // 碰撞中心点
    depth: f32,             // 穿透深度
    normal: Vector3<f32>,   // 碰撞法线
    timestamp: i64,
}
```

## 5. 开发步骤

1.  **原型验证 (Prototype)**:
    *   编写一个 `bin` 工具 `cargo run --bin check_collision -- --sys <system_name>`。
    *   实现 SQLite Self-Join 查询，获取候选列表。
    *   实现单线程的 Parry3D 检测。
2.  **并发优化**:
    *   引入 `rayon` 或 `tokio` 线程池进行并行计算。
3.  **结果持久化**:
    *   将结果存入 SurrealDB 或导出为 JSON/Parquet 供前端查看。
4.  **集成 API**:
    *   封装为 API `POST /api/spatial/collision_check`。
